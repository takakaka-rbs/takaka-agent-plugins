# /harvest-knowledge

ブランチ作業で得た学びを「次のセッションの自分（と各プラグイン）が使える形」で対象リポジトリに蓄積するスキルです。ブランチ作業の締め（コミット・プッシュの前後）に呼び出す使い方を想定していますが、セッション途中の単発の知見の記録にも使えます。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、ナレッジ候補の一覧・書き込み内容は報告のみで最後まで実行する
- 曖昧な点は合理的な仮定で進め、採用した仮定は最終報告に明記する
- **例外 — 次の操作だけは実行前に必ず確認する**: 既存ナレッジファイル・既存独自スキルの削除、既存内容の大部分を置き換える書き換え（追記・小幅更新は確認不要）

## Usage

```
/harvest-knowledge [対象リポジトリのパス] [補足（例: 今回のブランチの学びを蓄積 / VRT のベースライン運用だけ記録して）]
```

引数は省略可能です。省略した場合はカレントリポジトリを対象とし、範囲は実行前確認（ステップ0）で確認します。

## 全体フロー

### ステップ0: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。引数・プロジェクトの文脈から自明な項目は聞かずに省略し、全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **対象リポジトリ**: カレントディレクトリから判別できない場合のみ
2. **収穫範囲**: 現在ブランチの差分＋このセッションの作業内容（デフォルト提案）か、セッションの特定トピックのみか
3. **保存規約の初期化（対象リポジトリで初回のみ）**: 独自スキルは `.claude/skills/`、単発ナレッジは `docs/knowledge/`、索引は `CLAUDE.md` という配置でよいか。既に別の規約がある場合はそれに従う
4. **除外事項**: 記録してはいけない情報（未公開の意思決定・人事・認証情報まわりの経緯など）があるか

### ステップ1: 作業内容の収集

以下の材料からナレッジ候補を洗い出す（結果は候補一覧として報告し、続行する）。

- **ブランチ差分**: `git log` と `git diff <ベースブランチ>...HEAD` から、何を作り・何を変えたかを把握する（ベースブランチはリポジトリのフロー（dev / main）から判定する）
- **セッションの作業記録**: 試行錯誤の末に確立した手順、ハマった点とその原因・回避策、レビュー・テストで判明した落とし穴、調査して分かった対象リポジトリ固有の規約・制約
- **ユーザーの発言**: 作業中にユーザーが示した方針・好み・禁止事項

**ナレッジとみなす基準**（すべて満たすものだけ候補にする）:

- 次に似た作業をする人（または自分）の手戻りを減らす
- コード・設定・git 履歴を読むだけでは分からない（「なぜそうしたか」「何がダメだったか」を含む）
- 一過性でない（そのブランチ限りの事情・一時的なワークアラウンドの期限情報は、期限や条件を明記できる場合のみ候補にする）

**ナレッジにしないもの**: コードを読めば分かる実装内容、git 履歴・PR に既にある変更理由、公式ドキュメントの引き写し、対象リポジトリと無関係な一般知識。

### ステップ2: 既存ナレッジの調査（knowledge-researcher）

`knowledge-researcher` エージェントに対象リポジトリを調査させ、以下を把握する。

- 既存の独自スキル（`.claude/skills/`）・ナレッジMD（`docs/knowledge/`）・`CLAUDE.md` の索引と記述の現状
- ステップ1の各候補について、**新規作成 / 既存ファイルへの追記・更新 / 重複につき破棄**の推奨
- README・docs/ 配下の既存ドキュメントとの重複の有無

### ステップ3: 分類と書き込み設計

調査結果を踏まえ、各候補を以下の基準で振り分ける（設計は報告として提示し、続行する）。

| 候補の性質 | 保存先 | 形式 |
|---|---|---|
| 繰り返し実行する手順で、起動条件（トリガー）が言語化できる | `.claude/skills/<skill-name>/SKILL.md` | 独自スキル |
| 単発の知見・落とし穴・設計判断の記録 | `docs/knowledge/YYYY-MM-DD_<slug>.md` | ナレッジMD |
| 既存スキル・既存ナレッジの内容が古い/不足 | 既存ファイル | 追記・更新（新規は作らない） |
| リポジトリ全体の恒常的な規約で1〜2行で言い切れる | `CLAUDE.md` 本文の該当セクション | 直接追記 |

### ステップ4: 書き込みと索引メンテ

1. 設計どおりにファイルを作成・更新する（フォーマットは後述）
2. `CLAUDE.md` の `## 開発ナレッジ索引` セクション（無ければ末尾に作成）を更新する。**1件1行**で `- [タイトル](docs/knowledge/....md) — 一言のフック` の形式。独自スキルは自動読込されるため索引には載せない
3. 索引の肥大化を防ぐ: 本文・詳細は絶対に CLAUDE.md に書かない。索引が50行を超えたらカテゴリ分けを提案する（実施はユーザー判断）

### ステップ5: 最終報告

- 作成・更新したファイルの一覧（独自スキル / ナレッジMD / CLAUDE.md）
- 候補にしたが**除外したものとその理由**（重複・一過性・除外事項に抵触）
- 採用した仮定
- コミット・プッシュは行わず、`/git-commit-pusher` の利用を案内する

## ナレッジMDのフォーマット

```markdown
---
id: KNW-YYYY-MM-DD-<slug>
title: <一文のタイトル>
date: YYYY-MM-DD
branch: <収穫元ブランチ>
issues: ["#12"]
related: ["REQ-F-001", "API-003", "PBI-004"]
tags: [testing, ci]
---

## 事象 / 課題

（何をしようとして何が起きたか）

## 原因 / 背景

（コードを読むだけでは分からない理由）

## 対処 / 確立した手順

（次に同じ状況になったときにそのまま使える形で）

## 根拠

（該当ファイルのパス・PR・外部URL）
```

## 独自スキルのフォーマット

```markdown
---
name: <skill-name>
description: <三人称で「いつ使うか」を具体的なトリガー語とともに記述する。例: 「VRT のベースライン更新を求められたとき、"スクリーンショット差分" "toHaveScreenshot" に言及されたときに使用する」>
---

# <skill-name>

（手順を命令形で。対象リポジトリの実在コマンド・パスを使い、推測のコマンドは書かない）
```

## 品質基準

- **重複させない**: 既存に追記できるなら新規ファイルを作らない（ステップ2の調査結果に従う）
- **根拠を残す**: 各ナレッジに根拠（ファイルパス・PR・URL）を必ず付ける。確証のないことは確度を明記する
- **秘匿情報を書かない**: 認証情報・接続情報の値、ステップ0で指定された除外事項は記録しない
- **賞味期限を明記する**: バージョン依存・期限付きワークアラウンドは、無効になる条件を本文に書く
