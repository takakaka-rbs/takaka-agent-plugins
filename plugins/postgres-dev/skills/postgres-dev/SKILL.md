# /postgres-dev

PostgreSQL の DDL・DML を、対象リポジトリのスキーマ管理方式（マイグレーション運用）に沿って作成・変更するスキルです。要件定義書・API仕様書・実装計画からのテーブル設計にも、単発の「カラム追加」「サンプルデータ投入」にも使えます。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、テーブル設計・SQL は報告のみで最後まで実行する
- 作業中に判明した不明点は、調査・既存スキーマの類似定義で埋まらなければ合理的な仮定を置いて進める。採用した仮定は最終報告に明記する
- **例外 — 次の不可逆・破壊的操作だけは実行前に必ず確認する**: DROP TABLE / DROP COLUMN / TRUNCATE / データを失う型変更・制約変更を含むマイグレーションの作成と適用、ローカル・テスト用以外のDBへの適用、適用済みマイグレーションファイルの変更

## Usage

```
/postgres-dev [対象リポジトリのパス] [やりたいこと（例: users にメールアドレス列を追加 / 商品テーブルの新規設計 / サンプルデータ投入）]
```

引数は省略可能です。省略した場合は実行前確認（ステップ0）で確認します。

## 全体フロー

### ステップ0: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。引数・プロジェクトの文脈から自明な項目は聞かずに省略し、全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **対象リポジトリ**: カレントディレクトリから判別できない場合のみ
2. **変更内容と入力**: 何を作りたい/変えたいか、根拠となる文書（要件定義書・OpenSpec.yml・実装計画）の場所が引数から自明でない場合
3. **適用まで行うか**: マイグレーションファイルの作成まで／ローカル・テスト用DBへの適用と関連コード生成（jOOQ 等）の再実行まで、のどちらか（デフォルト提案: 適用まで）
4. **DML の扱い**: サンプル/シードデータの投入方式（マイグレーションに含める・別スクリプト・手動）に既存の慣習がなく判断が必要な場合

### ステップ1: スキーマ調査（postgres-researcher）

`postgres-researcher` エージェントに対象リポジトリを調査させ、以下を把握する（結果は報告として提示し、続行する）。

- スキーマ管理方式: マイグレーションツール（Flyway / Liquibase / sqldef 等）、マイグレーションファイルの配置・命名規約（例: `V<番号>__<説明>.sql`）・適用コマンド
- 既存スキーマ: 対象テーブルの現在の定義（既存マイグレーションの積み上げ結果）、命名規約（テーブル・カラム・制約・インデックスの慣習、snake_case 等）、共通カラム（created_at 等）の慣習
- コード生成: スキーマから生成されるコード（jOOQ / Prisma 等）の有無と再生成コマンド、生成物を手で編集してはならないこと
- DB接続・環境: ローカル/テスト用DBの起動方式（docker-compose / devcontainer）と接続情報の管理方式
- DML の慣習: シード/サンプルデータの既存の投入方式（例: `V1_1__insert_sample_user_data.sql` のようなマイグレーション同梱）

### ステップ2: 設計・報告

1. 変更内容をテーブル定義（テーブル・カラム・型・制約・インデックス・外部キー）として設計する。要件定義書・OpenSpec.yml が入力にある場合は、対応する REQ-F-XXX / API-XXX を設計の根拠として記録する
2. 型・制約は PostgreSQL の機能と既存スキーマの慣習に沿って選ぶ（不明点は既存定義の類似例 → PostgreSQL の一般的な慣習の順で判断し、仮定として注記する）
3. 設計（変更前後の定義・影響を受ける既存データの扱い）を**報告として**提示し、承認を待たずに続行する。破壊的変更が含まれる場合のみ例外規定に従って確認する

### ステップ3: マイグレーション作成

1. **適用済みマイグレーションは編集しない**。変更は常に新しいマイグレーションファイルとして、リポジトリの命名規約に従った次の連番で作成する
2. DDL は1マイグレーション1目的で作成する。DML（シード/サンプルデータ・データ移行）はリポジトリの慣習に従い、DDL と分離できる場合は別ファイルにする
3. データ移行を伴う変更（NOT NULL 追加・型変更等）は、既存データを壊さない手順（デフォルト値・段階的変更・バックフィル）で書く
4. ロールバック手順（逆方向の SQL、またはツールの undo 機構）をコメントまたは報告に含める

### ステップ4: 適用・検証（実行前確認で「適用まで」を選んだ場合）

1. ローカル/テスト用DBが起動していることを確認し、調査で判明した適用コマンド（例: `flyway migrate` 相当）を実行する
2. スキーマから生成されるコードがある場合は再生成コマンド（例: jOOQ codegen）を実行し、生成が通ることを確認する
3. 適用に失敗した場合は原因（SQL構文・既存データとの競合・環境）を切り分け、マイグレーションを修正して再適用する（適用前のファイル修正は破壊的操作にあたらない）
4. ビルドが通ることを確認する（生成コードの変更が既存コードに影響する場合、影響箇所を報告する。**既存アプリコードの修正はしない** — 必要なら `/code-implementer` を案内する）

### ステップ5: 完了報告

- 作成したマイグレーションファイルの一覧（パス・目的・対応する REQ-F-XXX / API-XXX）
- 変更前後のスキーマ定義の要約と、ロールバック手順
- 適用・コード生成・ビルドの実行結果（実施した場合）
- 採用した仮定の一覧と、未解決の課題（既存コードへの影響・ユーザー判断が必要な点）
- コミットは行っていないこと（`/git-commit-pusher` への案内）

## 制約

- ユーザーへの質問は実行前確認（ステップ0）の1回のみ。以降は報告のみで完遂する（例外規定の破壊的操作を除く）
- **適用済みマイグレーションファイルを変更・削除しない**（チェックサム不整合・環境間の差異を生むため。必要な場合は例外規定として必ず確認する）
- スキーマから生成されたコード（jOOQ 等）を手で編集しない（再生成コマンドで反映する）
- ローカル・テスト用以外のDB（本番・共有環境）に接続・適用しない
- 既存アプリケーションコードを修正しない（DDL/DML とコード生成の再実行までが本スキルの範囲）
- 接続情報・認証情報をマイグレーションファイル・レポートにハードコードしない
- リポジトリにマイグレーションツールが存在しない場合は、勝手に導入せず、導入の選択肢（Flyway 等）と手順を提案して素の SQL ファイルの作成までに留める
