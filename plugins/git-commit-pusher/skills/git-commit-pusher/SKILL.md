# /git-commit-pusher

未コミットの変更を内容ごとに分割し、Conventional Commits 準拠のコミットメッセージ（英語）でコミットしてプッシュするスキルです。

## ステップ0: 前提チェック（最優先・作業前に必ず実行）

分析を始める前に以下を確認し、**不足があればこの時点でユーザーに伝える**（後工程で気づくのは無駄なため）。

1. `git rev-parse --is-inside-work-tree` — gitリポジトリでなければ、その旨を伝えて終了する
2. `git remote -v` — リモートが未設定なら「コミットのみ可能でプッシュ不可」であることを伝え、コミットのみで続行するか確認する
3. `gh --version` と `gh auth status` — gh CLIが未インストールまたは未認証の場合はユーザーに伝える。ただしプッシュはgitの既存認証（credential manager等）で通ることも多いため、**警告のみで続行してよい**。プッシュが認証エラーになった時点で `gh auth login` などのセットアップを案内する
4. 未コミットの変更が1件もなければ（`git status --porcelain` が空）、その旨を伝えて終了する

## 全体フロー

### ステップ1: コミット規約の把握

1. リポジトリのcommitlint設定を探して読む（優先順: `commitlint.config.js` / `commitlint.config.mjs` / `.commitlintrc*` / `package.json` の `commitlint` フィールド）
2. 設定から以下を把握する
   - 許可されている `type` の一覧（`type-enum`）
   - ヘッダー最大文字数（`header-max-length`）
   - その他のルール（subject末尾ピリオド禁止など）
3. `.husky/commit-msg` の存在を確認し、huskyフックでcommitlintが強制されているかを把握する
4. 設定が見つからない場合は `@commitlint/config-conventional` の標準ルール（`type(scope): subject`、typeは小文字）に従う
5. 直近のコミット履歴（`git log --oneline -15`）を見て、scopeの使われ方・言語（既存が英語subjectか）の傾向を把握する

### ステップ2: 変更の把握

1. `git status --porcelain` で変更ファイルの全体像を取得する
2. `git diff` と `git diff --cached` で変更内容を確認する。新規ファイルは中身をReadで確認する
3. 既にステージ済みの変更がある場合は、その内容もコミット計画に含めて再構成することをユーザーに伝える（必要に応じて `git reset` でステージを一旦解除してから組み直す。ワークツリーの内容は変更しない）
4. 機密情報らしきファイル（`.env`、鍵ファイル、トークンを含むファイルなど）が変更に含まれる場合は警告し、コミット対象から除外することを提案する

### ステップ3: 変更のグループ分け

1. 変更を **論理的に独立した単位** に分割する。分割の観点:
   - 変更の種類（新機能 / バグ修正 / ドキュメント / 設定 など）→ `type` が異なるものは必ず別コミット
   - 対象領域（ディレクトリ・モジュール・プラグイン単位など）→ `scope` の候補
   - 依存関係（一方がないと動かない変更は同一コミットにまとめる）
2. 1ファイル内に複数の無関係な変更が混在する場合は、無理に分割せず（`git add -p` は対話的で使えない）、主たる変更のtypeでまとめ、コミット本文で補足する
3. グループごとにコミットメッセージを作成する。**メッセージは英語**で、ステップ1で把握したルールに従う
   - 形式: `type(scope): subject`（scopeは有用な場合のみ。既存履歴の慣習に合わせる）
   - typeは小文字、subjectは命令形・小文字始まり・末尾ピリオドなし
   - ヘッダーは最大文字数以内（既定100文字）
   - 変更の意図が一行で伝わらない場合は本文（body）を付ける。破壊的変更は `type!` と `BREAKING CHANGE:` フッターを使う

### ステップ4: コミット計画の提示・承認（必須）

1. 以下の形式でコミット計画を提示する

   ```
   コミット1: feat(scope): add xxx
     - path/to/file1
     - path/to/file2
   コミット2: docs: update yyy
     - path/to/file3
   プッシュ先: origin/<branch>
   ```

2. ユーザーに確認する
   - 分割単位・メッセージ・順序の修正有無
   - プッシュまで行うか、コミットのみで止めるか
3. **ユーザーの承認なしにコミット・プッシュを実行しない**

### ステップ5: コミット実行

1. 承認された計画に沿って、グループごとに `git add <対象ファイル>` → `git commit -m "<message>"` を実行する（本文がある場合は `-m` を2つ使うか一時ファイル経由で渡す）
2. huskyフック（commitlint）がコミットを拒否した場合は、エラー内容を読んでメッセージを修正し再試行する。**`--no-verify` でフックをバイパスしない**
3. フックがlint-staged等でファイルを自動整形した場合は、整形結果を確認して続行する
4. 途中で失敗した場合は、完了済みコミットと未処理の変更の状態を正確に報告する

### ステップ6: プッシュ

1. 現在のブランチを確認する。`main` / `master` への直接プッシュになる場合はその旨を明示して再確認する
2. upstream未設定のブランチは `git push -u origin <branch>` を使う
3. non-fast-forwardで拒否された場合は `git pull --rebase` を提案し、ユーザー承認のうえで実行する（**force pushはしない**）
4. 認証エラーの場合は `gh auth login` またはcredential設定を案内する

### ステップ7: 結果サマリー

- 作成したコミットの一覧（ハッシュ・メッセージ・ファイル数）
- プッシュ先（リモート・ブランチ）と結果
- スキップ・除外したファイルとその理由

## 制約

- **ユーザーの承認なしにコミット・プッシュしない**
- `--no-verify` などフックのバイパスを行わない（フック失敗はメッセージや内容を直して対処する）
- force push（`--force` / `--force-with-lease`）を行わない
- `git commit --amend` や rebase など履歴の書き換えを行わない（ユーザーが明示的に指示した場合を除く）
- ワークツリーのファイル内容を変更しない（コミット操作のみ。整形はフックに任せる）
- 機密情報らしきファイルは警告なしにコミットしない
- コミットメッセージに絵文字・日本語を使わない（英語・Conventional Commits準拠。ただし既存履歴が日本語運用ならユーザーに確認して合わせる）
