# /requirements-specificater

要件定義書（要件定義書テンプレートに準拠したMarkdown）を、ユーザーとの対話・コードベース調査・Web調査を通じて作成・更新するスキルです。

## Usage

```
/requirements-specificater [既存の要件定義書のパス | 要件メモ・ドラフトのテキスト]
```

引数は省略可能です。省略した場合は、新規作成か既存ファイルの更新かをユーザーに確認します。

## ベーステンプレート

`${CLAUDE_PLUGIN_ROOT}/docs/requirements-specification-template.md`

このテンプレートには、各セクションで何を記載すべきかを示すHTMLコメント（ガイド）と、
後続の画面仕様書・API仕様書プラグインとの連携を見据えたID採番ルールが含まれています。
作業前に必ずこのテンプレートを読み込み、構成・ID体系・コメントの内容を把握してください。

## 全体フロー

### ステップ1: 状況確認

1. `$ARGUMENTS` を確認する
   - 既存の要件定義書ファイルパスが指定された場合 → そのファイルを読み込み、続きから作業する
   - 要件メモ・テキストが渡された場合 → 新規作成として扱い、その内容を入力情報とする
   - 何も指定がない場合 → 「新規作成」か「既存ファイルの更新」かをユーザーに確認する
2. 出力先パスをユーザーに確認する。**毎回必ず確認し、固定パスを決め打ちしない**
   - 例として `docs/requirements/<project-name>.md` のような構成を提案してもよいが、最終的にはユーザーの指定に従う
3. 出力先にファイルが存在しない場合は、ベーステンプレートをコピーして作業ファイルとする。存在する場合はそれを読み込み、12章「未確定事項」と `_(未記入)_` 箇所から状況を把握する

### ステップ2: 初期情報のマッピング

1. ユーザーが渡した要件メモ・既存ドラフトの内容を、テンプレートの該当セクションへ転記する
2. 転記時はテンプレート内のHTMLコメント（各セクションのガイド）を参照し、内容を適切なセクションに分類する
3. この時点で `_(未記入)_` のまま残っている項目を洗い出し、ステップ3で扱うリストを作る

### ステップ3: 不足情報の調査・質問

1. 洗い出した未記入項目を次の3種類に分類する
   - **(a) コードベース調査で埋まる可能性がある項目**: 既存システム概要、技術スタック、既存API一覧、データモデルなど
   - **(b) Web調査で埋まる可能性がある項目**: 業界標準、外部サービスの仕様、一般的な非機能要件の目安など
   - **(c) ユーザーに確認するしかない項目**: ビジネス上の意思決定、優先度、予算・期限、ステークホルダー名など
2. (a)(b) は `requirements-researcher` エージェントに調査を依頼する
   - 1回の依頼に対象を詰め込みすぎず、関連する項目をまとめて渡す
   - 返ってきた「反映案」をテンプレートへ転記し、「残課題」は (c) のリストへ合流させる
3. (c) はユーザーに質問する
   - 一度に大量の質問をしない。優先度の高いもの（スコープ、機能要件、Mustの非機能要件など）から3〜5問程度ずつ確認する
   - 回答を得たら都度テンプレートへ反映する
   - 「わからない」「後で決める」という回答は、無理に埋めず12章「未確定事項」に `OPEN-XXX` として記録する

### ステップ4: ID採番・整合性チェック

1. 機能要件・非機能要件・画面・I/F・ユースケース・データエンティティ・リスク・オープンイシューに、
   テンプレートのID採番ルール（`REQ-F-XXX`, `REQ-NF-XXX`, `SCR-XXX`, `IF-XXX`, `UC-XXX`, `DAT-XXX`, `RISK-XXX`, `OPEN-XXX`）に従ってIDを割り当てる
2. 機能要件と画面ID・I/F IDの対応関係を確認する。対応する画面・I/Fが未定の場合は `TBD` のままにし、オープンイシューに記録する
3. **既存ファイルを更新する場合、既存のIDは変更・再採番しない**（後続の画面仕様書・API仕様書プラグインからの参照を壊さないため）。新規項目には欠番を避けて新しい連番を割り当てる

### ステップ5: 出力・サマリー

1. 完成または区切りのよい状態になったら、テンプレートのHTMLコメント（ガイド）を成果物から削除し、本文のみを残す
2. 出力先に保存する
3. ユーザーに以下をサマリーとして提示する
   - 完成度（埋まったセクション数 / 残っている `_(未記入)_` ・オープンイシューの件数）
   - 主要な機能要件・非機能要件の一覧
   - 未確定事項（`OPEN-XXX`）の一覧と、解決のために誰に何を確認すべきか
   - 次のステップの案内: 「この要件定義書の `SCR-XXX` / `IF-XXX` をもとに、画面仕様書プラグイン・API仕様書プラグイン（OpenSpec含む）で詳細化できます」

## 中断・再開

- 対話は複数回のセッションに分けてよい
- 再実行時は出力先ファイルを読み込み、12章のオープンイシューや `_(未記入)_` 箇所から再開する
- 1回のセッションで全項目を埋めきる必要はない。ユーザーが切り上げたい場合は、その時点の状態で保存し、残課題をオープンイシュー（`OPEN-XXX`）にまとめて終了する

## 制約

- ユーザーが確認すべき事項を、調査結果や推測だけで勝手に確定させない
- 機密情報・認証情報をWeb調査に渡さない（`requirements-researcher` エージェントへの依頼内容にも含めない）
- テンプレートの章構成・ID体系を変更しない（後続プラグインとの連携が壊れるため）
- 出力先パスは必ずユーザーに確認してから書き込む
