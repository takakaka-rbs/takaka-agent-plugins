# /requirements-specificater

要件定義書（要件定義書テンプレートに準拠したMarkdown）を、ユーザーとの対話・コードベース調査・Web調査を通じて作成・更新するスキルです。

## この文書が扱う抽象度（重要）

要件定義書は「これから作るサービスがどんな要件を持つのか」を示す文書です。

- 書くこと: 「本サービスは、ユーザーの情報を特定するためのログイン機能を持つ。この機能は以下の要件を満たす」といった抽象度の要件
- 書かないこと: 具体的な画面レイアウト・画面遷移・APIエンドポイント・データ設計。これらは本書の要件ID（REQ-F-XXX）を参照して、画面仕様書・OpenSpec.yml（API仕様書）として別途作成する
- 参照方向は常に「仕様書 → 要件定義書」の一方向。要件定義書の側に画面番号・API番号は書かない

ユーザーが具体的な画面やAPIの話を始めた場合は、その内容から「満たすべき要件」を抽出して記録し、具体的な仕様は画面仕様書・OpenSpec.yml側で扱う旨を案内する。

## Usage

```
/requirements-specificater [既存の要件定義書のパス | 要件メモ・ドラフトのテキスト]
```

引数は省略可能です。省略した場合は、新規作成か既存ファイルの更新かをユーザーに確認します。

## ベーステンプレート

`${CLAUDE_PLUGIN_ROOT}/docs/requirements-specification-template.md`

このテンプレートには、文書の位置づけ・書き方の指針・ID採番ルール・各セクションの記入ガイド（HTMLコメント）が含まれています。作業前に必ずこのテンプレートを読み込み、構成・抽象度・ID体系を把握してください。

## 全体フロー

### ステップ1: 状況確認

1. `$ARGUMENTS` を確認する
   - 既存の要件定義書ファイルパスが指定された場合 → そのファイルを読み込み、続きから作業する
   - 要件メモ・テキストが渡された場合 → 新規作成として扱い、その内容を入力情報とする
   - 何も指定がない場合 → 「新規作成」か「既存ファイルの更新」かをユーザーに確認する
2. 出力先パスをユーザーに確認する。**毎回必ず確認し、固定パスを決め打ちしない**
   - 例として `docs/requirements/<service-name>.md` のような構成を提案してもよいが、最終的にはユーザーの指定に従う
3. 出力先にファイルが存在しない場合は、ベーステンプレートをコピーして作業ファイルとする。存在する場合はそれを読み込み、6章「未確定事項」と `_(未記入)_` 箇所から状況を把握する

### ステップ2: 初期情報のマッピング

1. ユーザーが渡した要件メモ・既存ドラフトの内容を、テンプレートの該当セクションへ転記する
2. 転記時は抽象度に注意する
   - 画面・API・実装の詳細が書かれていたら、そこから「満たすべき要件」を抽出して機能要件として記録する（詳細そのものは転記しない）
   - 機能ごとに「本サービスは〜する機能を持つ」の形でまとめ、その下に要件を列挙する
3. この時点で `_(未記入)_` のまま残っている項目を洗い出し、ステップ3で扱うリストを作る

### ステップ3: 不足情報の調査・質問

1. 洗い出した未記入項目を次の3種類に分類する
   - **(a) コードベース調査で埋まる可能性がある項目**: 既存サービスの背景、現行機能の把握、既存の制約条件など
   - **(b) Web調査で埋まる可能性がある項目**: 業界標準、類似サービスの一般的な機能構成、非機能要件の一般的な水準など
   - **(c) ユーザーに確認するしかない項目**: サービスの目的、スコープ、優先度、ビジネス上の意思決定など
2. (a)(b) は `requirements-researcher` エージェントに調査を依頼する
   - 1回の依頼に対象を詰め込みすぎず、関連する項目をまとめて渡す
   - 返ってきた「反映案」をテンプレートへ転記し、「残課題」は (c) のリストへ合流させる
3. (c) はユーザーに質問する
   - 一度に大量の質問をしない。優先度の高いもの（目的、スコープ、主要な機能要件など）から3〜5問程度ずつ確認する
   - 回答を得たら都度テンプレートへ反映する
   - 「わからない」「後で決める」という回答は、無理に埋めず6章「未確定事項」に `OPEN-XXX` として記録する
4. 図・画像で理解が深まる箇所（サービス全体像、業務フローなど）には、Mermaid図などの埋め込みを提案する

### ステップ4: ID採番・整合性チェック

1. 機能要件・非機能要件・ユースケース・オープンイシューに、テンプレートのID採番ルール（`REQ-F-XXX`, `REQ-NF-XXX`, `UC-XXX`, `OPEN-XXX`）に従ってIDを割り当てる。`REQ-F-XXX` は機能をまたいで通しの連番にする
2. 各機能要件が「実現手段に踏み込まない抽象度」と「達成判断ができる具体度」を両立しているか確認する。画面名・エンドポイント・テーブル名が混入していたら要件の言葉に置き換える
3. 各機能要件がいずれかのユースケース（UC-XXX）と対応しているか確認する。対応がない要件は、ユースケースの追加が必要か、要件自体が不要かをユーザーに確認する
4. **既存ファイルを更新する場合、既存のIDは変更・再採番しない**（画面仕様書・OpenSpec.yml からの参照を壊さないため）。新規項目には欠番を避けて新しい連番を割り当てる

### ステップ5: 出力・サマリー

1. 完成または区切りのよい状態になったら、テンプレートのHTMLコメント（記入ガイド・記入例）を成果物から削除し、本文のみを残す
2. 出力先に保存する
3. ユーザーに以下をサマリーとして提示する
   - 完成度（埋まったセクション / 残っている `_(未記入)_` ・オープンイシューの件数）
   - 機能要件の一覧（機能ごと）と主要な非機能要件
   - 未確定事項（`OPEN-XXX`）の一覧と、解決のために誰に何を確認すべきか
   - 次のステップの案内: 「この要件定義書の `REQ-F-XXX` を参照して、画面仕様書プラグイン・API仕様書プラグイン（OpenSpec.yml）で具体的な画面・APIを定義できます」

## 中断・再開

- 対話は複数回のセッションに分けてよい
- 再実行時は出力先ファイルを読み込み、6章のオープンイシューや `_(未記入)_` 箇所から再開する
- 1回のセッションで全項目を埋めきる必要はない。ユーザーが切り上げたい場合は、その時点の状態で保存し、残課題をオープンイシュー（`OPEN-XXX`）にまとめて終了する

## 制約

- 画面レイアウト・画面遷移・APIエンドポイント・データ設計を要件定義書に書かない（それぞれ画面仕様書・OpenSpec.ymlの担当）
- スケジュール・リスク管理・受け入れテスト基準も本書では扱わない（必要なら別文書を提案する）
- ユーザーが確認すべき事項を、調査結果や推測だけで勝手に確定させない
- 機密情報・認証情報をWeb調査に渡さない（`requirements-researcher` エージェントへの依頼内容にも含めない）
- テンプレートの章構成・ID体系を変更しない（画面仕様書・API仕様書プラグインとの連携が壊れるため）
- 出力先パスは必ずユーザーに確認してから書き込む
