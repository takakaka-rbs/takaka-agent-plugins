<!--
  要件定義書テンプレート（requirements-specificater プラグイン用）

  ## この文書の位置づけ

  この要件定義書は「これから作るサービスがどんな要件を持つのか」を示す文書です。
  具体的な画面レイアウトやAPI仕様はここには書きません。それらは本書の要件IDを
  参照しながら、それぞれ画面仕様書・OpenSpec.yml（API仕様書）として別途作成します。

    要件定義書（本書）          … サービスが満たすべき要件を抽象的に定義する
      ↑ REQ-F-XXX / REQ-NF-XXX を参照
    画面仕様書                  … 要件を満たす具体的な画面を定義する
    OpenSpec.yml（API仕様書）   … 要件を満たす具体的なAPIを定義する

  参照は常に「仕様書 → 要件定義書」の一方向です。本書の側に画面番号・API番号を
  書く必要はありません。

  ## 書き方の指針

  - 要件は「本サービスは〜する機能を持つ。その機能は以下の要件を満たす」という
    抽象度で書く。実現手段（特定の画面・エンドポイント・テーブル設計）には踏み込まない
  - 「何を満たせば要件達成か」が判断できる粒度までは具体化する
    （例: ×「ログインできる」 ○「登録済みユーザーを一意に特定し、本人以外の
    なりすましを防げる」）
  - 理解を助けるための図・画像（Mermaid図、概念図、業務フロー図など）の埋め込みは推奨
  - 未記入の項目には `_(未記入)_` を残し、確定した時点で本文に置き換える。
    `_(未記入)_` のまま完成扱いにしない
  - スケジュール・リスク管理・受け入れテスト基準・データ設計は本書の管理対象外。
    必要であれば別文書として作成する

  ## ID 採番ルール

  - REQ-F-XXX  : 機能要件（画面仕様書・OpenSpec.yml が参照するキー）
  - REQ-NF-XXX : 非機能要件
  - UC-XXX     : ユースケース
  - OPEN-XXX   : 未確定事項（オープンイシュー）

  XXX は 001 から始まる3桁の連番（必要なら桁を増やす）。
  既存の要件定義書を更新する場合、既存IDは変更せず欠番のまま維持すること
  （画面仕様書・OpenSpec.yml からの参照が壊れるため）。

  各セクションの <!-- ... --> コメントは記入ガイドです。
  最終成果物ではコメントを削除し、本文のみを残してください。
-->
