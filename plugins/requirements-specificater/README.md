# requirements-specificater

コードベース調査・Web調査を通じて、「これから作るサービスがどんな要件を持つのか」を示す要件定義書を作成・更新するプラグインです。質問はタスク開始時の実行前確認1回のみで、以降は自動で完遂します。

## 概要

ユーザーが要件メモやドラフトを書くと、`/requirements-specificater` スキルが以下を行います。

1. **実行前確認（唯一の質問タイミング）**: 新規/更新・出力先・対象スコープなど方針判断が必要な事項を、最大4問まで最初に1回だけ確認する（文脈から自明な項目は聞かない）
2. 内容を要件定義書テンプレートの各セクションに整理する
3. 不足している情報を「コードベース調査」「Web調査」「合理的な仮定（注記 + `OPEN-XXX` 記録）」で埋める
4. 機能要件・非機能要件・ユースケースにIDを採番する
5. テンプレートに沿った要件定義書を完成させ、採用した仮定と未確定事項を最終報告で開示する

## 文書の役割分担

要件定義書は要件を **抽象的に** 定義する文書です。具体的な画面やAPIの仕様は書きません。

```
要件定義書（本プラグイン）        … サービスが満たすべき要件を定義する
  ↑ REQ-F-XXX / REQ-NF-XXX を参照
画面仕様書（別プラグイン・予定）   … 要件を満たす具体的な画面を定義する
OpenSpec.yml（別プラグイン・予定） … 要件を満たす具体的なAPIを定義する
```

要件の書き方の例:

> 本サービスは、ユーザーの情報を特定するためのログイン機能を持つ。
> この機能は以下の要件を満たす。
>
> - REQ-F-001: 登録済みユーザーを一意に特定できる（Must）
> - REQ-F-002: 本人以外によるなりすましログインを防止できる（Must）

このログイン機能を実現する具体的な画面・APIは、画面仕様書・OpenSpec.yml 側が
`REQ-F-XXX` を参照しながら定義します。参照は常に「仕様書 → 要件定義書」の一方向で、
要件定義書の側に画面番号・API番号は書きません。

また、スケジュール・リスク管理・受け入れテスト基準・データ設計も要件定義書の管理対象外です
（必要な場合は別文書として作成します）。

## 構成要素

| コンポーネント | ファイル | 説明 |
|---|---|---|
| Skill | `skills/requirements-specificater/SKILL.md` | `/requirements-specificater` スラッシュコマンド。実行前確認から出力までの全体フローを制御する |
| Agent | `agents/requirements-researcher.md` | コードベース調査・Web調査を担当するサブエージェント |
| テンプレート | `docs/requirements-specification-template.md` | 要件定義書のベーステンプレート（ID採番ルール・記入ガイド付き） |

## ID採番ルール

| プレフィックス | 対象 | 説明 |
|---|---|---|
| `REQ-F-XXX` | 機能要件 | 本書の中心。画面仕様書・OpenSpec.yml が参照するキーになる |
| `REQ-NF-XXX` | 非機能要件 | 性能・信頼性・セキュリティなど、サービス全体の品質要件 |
| `UC-XXX` | ユースケース | ユーザーがサービスで達成したいゴール。機能要件の根拠になる |
| `OPEN-XXX` | 未確定事項 | 要件定義フェーズの残課題。仕様書作成前に解消推奨 |

`XXX` は `001` から始まる3桁の連番です。**既存の要件定義書を更新する際、既存IDは変更しません**
（画面仕様書・OpenSpec.yml からの参照が壊れるため）。新規項目には新しい連番を割り当てます。

## 使い方

```
/requirements-specificater
```

新規作成・既存ファイルの更新・出力先パスなどは、実行前確認（最大4問・自明な項目は省略）で最初に1回だけ確認し、以降は質問せずに完遂します。

既存の要件定義書を更新する場合はパスを引数に渡せます。

```
/requirements-specificater docs/requirements/my-service.md
```

要件メモをそのまま渡して開始することもできます。

```
/requirements-specificater 在庫管理システムのリプレイス。既存システムはExcel運用。
```

## 今後の連携予定

このプラグインが出力する要件定義書をインプットに、以下のプラグインが連携します。

- **API仕様書プラグイン（[openapi-specificater](../openapi-specificater/README.md)・作成済み）**: 本書の `REQ-F-XXX` を参照して、要件を満たすAPIをOpenSpec.ymlとして定義
- **画面仕様書プラグイン（作成予定）**: 本書の `REQ-F-XXX` を参照して、要件を満たす画面のレイアウト・入力項目・遷移を定義

そのため、本プラグインの章構成・ID採番ルールは変更時に互換性に注意してください。

## インストール

```bash
PLUGIN=requirements-specificater
PROJECT=/path/to/your/project

cp -r agents/* $PROJECT/.claude/agents/
cp -r skills/* $PROJECT/.claude/skills/
```

## 更新履歴

### 0.2.0

- 対話フローを「初回一括確認・以降は自動完遂」に変更: 実行前確認（最大4問・自明な項目は省略）を新設し、作業中のユーザーへの質問を廃止
- ユーザーが決める項目は合理的な仮定（「（仮定）」注記 + `OPEN-XXX` 記録）で埋め、採用した仮定を最終報告に明記するポリシーを追加
- 例外規定を追加: 既存ファイルの削除など不可逆な破壊的操作のみ実行前に必ず確認
