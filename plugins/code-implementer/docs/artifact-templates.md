# code-implementer 成果物テンプレート

エージェント間で受け渡す成果物ファイルの必須構成を定義する。各エージェントはここに定義された見出し構成を**厳守**する（下流エージェントが見出しを頼りに読み取るため）。セクションを省略する場合は見出しを残して「該当なし」と書く。

配置先はすべて `<対象リポジトリ>/.claude/code-implementer/` 配下（[architecture.md](architecture.md) 参照）。

---

## 1. 調査レポート（research-report.md）

出力者: `dev-researcher` / 読者: 全エージェント

```markdown
# 調査レポート: <対象リポジトリ名>

- 作成日: YYYY-MM-DD
- 実装スコープ: <対象の API-XXX / SCR-XXX>

## 1. リポジトリプロファイル

### 1.1 技術スタック

（言語・フレームワーク・主要ライブラリとバージョン。根拠となるビルドファイル・lockファイルのパスを併記）

### 1.2 ディレクトリ構成とレイヤー構成

（バックエンド・フロントエンドそれぞれの構成。レイヤーの責務と配置ルールを既存コードの実例パスつきで）

### 1.3 命名・コーディング規約

（パッケージ/ディレクトリ・クラス/コンポーネント・メソッドの命名規約。lint 設定の場所。既存コードの実例つき）

### 1.4 DBアクセス方式

（ORマッパー等の種別・利用方法の実例・スキーマ管理方式（マイグレーションツール）・コード生成の有無）

### 1.5 コード生成

（OpenAPI Generator 等の生成対象・生成コマンド・生成物の配置先。生成されたコードは手で編集しないこと、を明記）

### 1.6 実行コマンド

| 用途 | コマンド | 実行ディレクトリ | 根拠 |
|---|---|---|---|
| ビルド | | | （Makefile / package.json 等の定義箇所。実在確認できないものは「未確認」と明記） |
| lint | | | |
| 単体テスト（バックエンド） | | | |
| 単体テスト（フロントエンド） | | | |
| アプリ起動（バックエンド） | | | |
| アプリ起動（フロントエンド） | | | |
| DBマイグレーション | | | |

### 1.7 テスト環境

（テスト用DBの構成（docker-compose / devcontainer / インメモリ）、テスト時のプロファイル・設定ファイル）

## 2. 実装方式調査

### 2.1 対象機能の実現方式

（実装スコープの各 API-XXX / SCR-XXX を当該スタックでどう実現するか。既存の類似実装があればそのパスを最優先の根拠として示す）

### 2.2 参考情報

（Web調査の出典。確度（コードから確認 / 一般論 / 推測）を必ず付記）

## 3. 残課題

（調査で解決しなかった点。下流エージェントが仮定を置く必要がある箇所）
```

## 2. 実装計画（implementation-plan.md）

出力者: `dev-planner`（修正ループで計画に影響する修正が出た場合の改訂も、`fix-planner` の修正計画を入力に `dev-planner` が行う）/ 読者: dev-backend, dev-frontend, テストエージェント

```markdown
# 実装計画: <対象機能>

- 作成日: YYYY-MM-DD
- 入力仕様書: <OpenSpec.yml / 画面仕様書のパス>
- 調査レポート: research-report.md

## 1. タスク一覧

| タスクID | 担当 | 対応仕様 | 対象ファイル | 依存タスク | 概要 |
|---|---|---|---|---|---|
| TASK-001 | backend | API-001 | （新規/変更するファイルのパス） | なし | |
| TASK-002 | frontend | SCR-001 | | TASK-001 | |

（タスクIDは TASK-001 からの連番。担当は backend / frontend のいずれか）

## 2. 実装順序

（依存関係に基づく実行順。並行可能なタスクがあれば明記）

## 3. 影響範囲

## 5. 改訂履歴
```

## 3. テスト結果レポート（test-reports/<level>-attempt<N>.md）

```markdown
## 1. サマリー

| 結果 | PASS / FAIL / ERROR（実行自体が失敗） / SKIP（ツール未導入等で実行対象外） |

| スキップ | |
  ```

## 4. 修正計画（fix-plans/fix-plan-attempt<N>.md）

出力者: `fix-planner` / 読者: dev-backend, dev-frontend, テストエージェント（スキルが該当タスクを振り分ける）

```markdown
# 修正計画 (attempt <N>)

- 作成日時: YYYY-MM-DD HH:MM
- 分析対象レポート: test-reports/<...>.md

## 1. 失敗原因の分類サマリー

| 失敗ケース | 分類 | 担当 |
|---|---|---|
| FAIL-1 | 実装起因 / テスト起因 / 仕様起因 / 環境起因 | dev-backend / dev-frontend / test-<level> / （対象外） |

## 2. 修正タスク

（分類が「実装起因」「テスト起因」のケースのみ。1タスク1見出し）

### FIX-1: <概要>

- 対象失敗ケース: FAIL-1
- 担当エージェント: dev-backend / dev-frontend / test-<level>
- 原因分析: （なぜ失敗したか。根拠となるコード箇所）
- 修正対象ファイル: <パス>
- 修正内容: （何をどう変えるか）
- 修正後に再実行するテストレベル: <level>

## 3. コード修正の対象外

- 仕様起因: （該当ケースと、仕様書側で解決すべき内容。/openapi-specificater 等への案内）
- 環境起因: （該当ケースと、ユーザーが行うべき環境修正の手順）

## 4. 再実行指示

- 再実行するテストレベルと順序: 
- リトライカウント: <このレベルの実装起因リトライが何回目か> / 上限
```
