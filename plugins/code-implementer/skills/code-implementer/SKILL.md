# /code-implementer

API仕様書（OpenSpec.yml の API-XXX）・画面仕様書（SCR-XXX）を入力に、対象リポジトリのバックエンド（Spring）・フロントエンド（Vue3）を自動実装し、テストと修正ループまで進行管理するスキルです。

あなた（メイン会話）はこのワークフローのオーケストレーターです。実装・テストの実作業はすべてサブエージェントに委譲し、自分では成果物ファイルの受け渡し・順序制御・ループ判定・報告だけを行います。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、計画・テスト結果は報告のみで最後まで実行する
- 作業中に判明した不明点は、調査・既存コードの類似実装で埋まらなければ合理的な仮定を置いて進める。採用した仮定は最終報告に明記する
- **例外 — 次の不可逆・破壊的操作だけは実行前に必ず確認する**: 既存の公開APIインターフェース・DBスキーマへの破壊的変更、既存ファイルの削除、実装計画に含まれない既存コードの大規模な書き換え

## 前提と設計ドキュメント

- ワークフロー設計・エージェント責務: `${CLAUDE_PLUGIN_ROOT}/docs/architecture.md`
- 成果物（調査レポート・実装計画・テスト結果・修正計画）の形式: `${CLAUDE_PLUGIN_ROOT}/docs/artifact-templates.md`

エージェントへの委譲時は、該当テンプレートの見出し構成に従うことを毎回指示してください。

## Usage

```
/code-implementer [対象リポジトリのパス] [仕様書のパス | 実装対象の API-XXX / SCR-XXX]
```

引数は省略可能です。省略した場合は実行前確認（ステップ0）で確認します。

## 成果物の受け渡し

エージェント間の受け渡しは `<対象リポジトリ>/.claude/code-implementer/` 配下のファイルを正とします。

```
.claude/code-implementer/
├── research-report.md          # dev-researcher の出力
├── implementation-plan.md      # dev-planner の出力
├── test-reports/<level>-attempt<N>.md
└── fix-plans/fix-plan-attempt<N>.md
```

各エージェントには常に「入力ファイルのパス」と「出力先パス」を明示して委譲します。作業ディレクトリが対象リポジトリの Git 管理に含まれる場合は、`.gitignore` への `.claude/code-implementer/` 追記を初回に**報告**します（勝手にコミットしない）。

## 全体フロー

### ステップ0: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。引数・プロジェクトの文脈から自明な項目は聞かずに省略し、全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **対象リポジトリ**: カレントディレクトリが対象リポジトリと判別できない場合のみ
2. **入力仕様書と実装スコープ**: OpenSpec.yml・画面仕様書の場所、実装対象の API-XXX / SCR-XXX の範囲が引数から自明でない場合。仕様書が見つからない場合は「場所の指定」または「対象外として進める」（バックエンドのみ・フロントエンドのみの実装になる）ことを確認する
3. **実行するテストレベル**: unit-backend / unit-frontend / api / vrt / e2e から選択（デフォルト提案: unit-backend + unit-frontend + api。vrt / e2e はツール導入状況が不明な場合その旨を添える）
4. **修正ループの最大リトライ回数**: デフォルト3回（テストレベルごとの実装起因リトライの上限）

### ステップ1: 調査（dev-researcher）

1. `dev-researcher` に対象リポジトリのパス・実装スコープ・出力先（`research-report.md`）を渡して委譲する
2. 返ってきた要点（スタック・特記すべき規約・残課題）を**報告として**提示し、続行する
3. 調査で「サポート対象外のスタック」（Spring / Vue3 以外が主体）と判明した場合は、その旨と品質リスクを報告した上で続行する（打ち切りはユーザーの指示があった場合のみ）

### ステップ2: 実装計画（dev-planner）

1. `dev-planner` に調査レポート・仕様書のパス・実装スコープ・出力先（`implementation-plan.md`）を渡して委譲する
2. 計画のタスク一覧・実装順序・影響範囲を**報告として**提示し、承認を待たずに続行する
3. 計画に**破壊的変更**が含まれる場合のみ、例外規定に従いその項目だけユーザーに確認する

### ステップ3: 実装（dev-backend / dev-frontend）

1. 実装計画のタスクを依存関係の順に実行する。各タスクを担当エージェント（backend → `dev-backend`、frontend → `dev-frontend`）に、タスクID・調査レポート・実装計画・仕様書のパスを渡して委譲する
2. 各エージェントの完了報告（変更ファイル・検証結果・仮定・申し送り）を確認する。ビルド失敗で返ってきた場合は、エラー内容を添えて同じエージェントに1回だけ再委譲し、それでも失敗するなら修正ループ（ステップ5）と同様に `fix-planner` に分析させる
3. 「生成元（openapi.yml / DDL）の更新が必要」という申し送りが来た場合: 生成元の変更が実装計画にあるタスクなら計画に従って実行し、計画にない場合は仕様起因の課題として記録して当該タスクをスキップする

### ステップ4: テスト（テストエージェント）

1. ステップ0で選択されたテストレベルを次の順に実行する: `unit-backend` → `unit-frontend` → `api` → `vrt` → `e2e`
2. 各レベルの担当エージェント（`test-unit-backend` / `test-unit-frontend` / `test-api` / `test-vrt` / `test-e2e`）に、テスト対象・調査レポート・仕様書のパス・レポート出力先（`test-reports/<level>-attempt<N>.md`）・attempt 番号を渡して委譲する
3. レベルが PASS したら次のレベルへ進む。FAIL / ERROR ならそのレベルで停止してステップ5（修正ループ）に入る（後段レベルは修正後に実行する）
4. `SKIP`（VRTツール未導入等）は導入案内を記録して次のレベルへ進む

### ステップ5: 修正ループ（fix-planner → dev-backend / dev-frontend → 再テスト）

1. `fix-planner` に失敗レポート・調査レポート・実装計画・仕様書のパス・リトライ状況・出力先（`fix-plans/fix-plan-attempt<N>.md`）を渡して分析・修正計画の作成を委譲する
2. 修正計画の分類に従って再作業を振り分ける:
   - **実装起因** → 該当 FIX を `dev-backend` / `dev-frontend` に委譲（このレベルのリトライカウント +1）
   - **テスト起因** → 該当 FIX を当該テストエージェントに委譲（リトライカウントに数えない）
   - **仕様起因** → コード修正せず、仕様側の課題として記録して続行（該当ケースは以後の判定から除外する）
   - **環境起因** → 環境修正の手順を記録し、当該レベルを SKIP 扱いにして続行
3. 再作業後、修正計画の「再実行指示」に従って該当テストレベルを attempt を進めて再実行する
4. **終了条件**（いずれかで修正ループを打ち切る）:
   - 対象レベルが PASS → ステップ4の次のレベルへ
   - 同一レベルの実装起因リトライがステップ0の上限（デフォルト3回）に到達 → 未解決として記録し、残りのレベルは実行せずステップ6へ
   - fix-planner が打ち切りを推奨（修正の見込みなし・同じ失敗の繰り返し）→ 未解決として記録しステップ6へ
   - 安全弁: ワークフロー全体の実装起因リトライ合計が「上限 × 選択レベル数」を超過 → 全体を打ち切りステップ6へ

### ステップ6: 完了報告

ユーザーに以下をサマリーとして提示する。

- 実装したタスク一覧（TASK-XXX / 対応 API-XXX・SCR-XXX / 作成・変更ファイル）
- テスト結果（レベルごとの最終 attempt の PASS/FAIL/SKIP と件数、レポートのパス）
- 修正ループの履歴（attempt 数・主な修正内容）
- **採用した仮定の一覧**(調査・計画・実装・テストで置いたもの)
- 未解決の課題: 仕様起因（`/openapi-specificater` / `/screen-specificater` への案内を含む）・環境起因（ユーザーが行う修正手順）・リトライ上限で残った失敗
- コミットは行っていないこと（コミットは `/git-commit-pusher` に委ねる旨の案内）

## 中断・再開

- 作業は複数セッションに分けてよい。再実行時も質問は実行前確認の1回のみ
- 再実行時は `.claude/code-implementer/` 配下の成果物（調査レポート・実装計画・最後の attempt のレポート・fix-plan）を読み込み、続きから再開する（調査・計画が既にあれば作り直さない。対象リポジトリの構成が大きく変わっている場合のみ dev-researcher に差分調査させる）

## 制約

- ユーザーへの質問は実行前確認（ステップ0）の1回のみ。以降は報告のみで完遂する（例外規定の破壊的操作を除く）
- メイン会話（あなた）は対象リポジトリのソースコードを直接編集しない。実装・テスト・分析はすべて担当エージェントに委譲する（成果物ディレクトリの作成・軽微なファイル操作は可）
- git のコミット・プッシュ・ブランチ操作をしない（`/git-commit-pusher` の担当。完了報告で案内する）
- 仕様書（OpenSpec.yml・画面仕様書・要件定義書）を変更しない。仕様の不備は課題として記録し、該当スキルでの修正を案内する
- 各エージェントの制約（実装エージェントはテストを書かない、テストエージェントは実装を直さない等）を跨ぐ指示を出さない
- テストが失敗したまま・チェックリスト未達のまま「完成」と報告しない（未解決事項は必ずサマリーで開示する）
- 機密情報（認証情報・接続文字列）をエージェントへの指示・Web調査に含めない
