# /backlog-manager

EPIC（大きな目標）と PBI（プロダクトバックログアイテム）を、対象リポジトリの `docs/backlog/` 配下のマークダウンとして管理するスキルです。新規起票にも、既存アイテムの更新・ステータス変更にも使えます。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、ドラフトは報告のみで最後まで実行する
- 曖昧な点は合理的な仮定で進め、採用した仮定は最終報告に明記する
- **例外 — 次の操作だけは実行前に必ず確認する**: EPIC・PBI ファイルの削除、採番済み ID の変更（ID は Issue・ブランチ・仕様書から参照されうる公開インターフェースとして扱う）

## Usage

```
/backlog-manager [対象リポジトリのパス] [やりたいこと（例: 要件定義書から PBI を起票 / PBI-004 を done に / 通知機能の EPIC を作成）]
```

引数は省略可能です。省略した場合はカレントリポジトリを対象とし、内容は実行前確認（ステップ0）で確認します。

## 全体フロー

### ステップ0: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。引数・プロジェクトの文脈から自明な項目は聞かずに省略し、全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **対象リポジトリ**: カレントディレクトリから判別できない場合のみ
2. **作業内容**: 何を作る/変えるか（EPIC 起票 / PBI 起票 / ステータス・内容の更新）が引数から自明でない場合
3. **入力ソース**: 起票の根拠（要件定義書のパス・自由記述・このセッションの文脈）が自明でない場合
4. **Issue 化まで行うか**: 起票した PBI を続けて GitHub Issue にするか（デフォルト提案: ready にした PBI のみ Issue 化）

### ステップ1: 既存バックログの調査（backlog-researcher）

`backlog-researcher` エージェントに対象リポジトリを調査させ、以下を把握する（結果は報告として提示し、続行する）。

- `docs/backlog/` の現状（配置・索引・**次に使うべき EPIC / PBI の連番**）
- 要件定義書の REQ-F-XXX 一覧と、既存 PBI によるカバー状況（重複起票の防止）
- 関連する既存 GitHub Issue・作業ブランチ

初回（`docs/backlog/` が存在しない場合）は、後述のディレクトリ構成で初期化する。

### ステップ2: ドラフト作成

調査結果を踏まえ、EPIC / PBI のドラフトを作成する（ドラフトは報告として提示し、続行する）。

- ID は既存の連番の続きから採番する（EPIC-001, PBI-001, ...）
- PBI は必ず親 EPIC を持つ。適切な EPIC がなければ EPIC の新設をドラフトに含める
- 要件定義書がある場合、PBI の `requirements` に対応する REQ-F-XXX を必ず記録する（対応がない PBI は「要件外」の理由を本文に書く）
- 受け入れ条件はチェックリスト形式で、**検証可能な文**（実行して真偽が判定できる）だけを書く
- 1 PBI は 1 ブランチ / 1 PR で完結する粒度を目安にし、大きすぎる場合は分割してドラフトする

### ステップ3: 書き込みと索引メンテ

1. `docs/backlog/epics/` / `docs/backlog/pbis/` にファイルを作成・更新する（フォーマットは後述）
2. `docs/backlog/README.md`（索引）を更新する: EPIC ごとに配下 PBI の一覧表（ID・タイトル・status・Issue）を保つ
3. ステータス更新の場合は frontmatter の `status` と `updated` を変更し、完了時は関連 Issue・PR 番号を記録する

### ステップ4: Issue 化（実行前確認で選択した場合のみ）

1. ready にした PBI を `/github-issue-creator` スキルに委譲して GitHub Issue を作成する（PBI の本文・受け入れ条件をそのまま Issue ボディの素材として渡す）
2. 作成された Issue 番号を PBI の frontmatter `issues` に書き戻す

### ステップ5: 最終報告

- 作成・更新したファイルの一覧（EPIC / PBI / 索引）
- ID の対応表（PBI ↔ REQ-F ↔ Issue）
- 採用した仮定・分割の判断
- コミット・プッシュは行わず、`/git-commit-pusher` の利用を案内する

## ディレクトリ構成（対象リポジトリ側）

```
docs/backlog/
├── README.md                      # 索引: EPIC ごとの PBI 一覧とステータス
├── epics/
│   └── EPIC-001_user-auth.md
└── pbis/
    └── PBI-001_login-screen.md
```

## EPIC のフォーマット

```markdown
---
id: EPIC-001
title: <ユーザーに提供する大きな価値を一文で>
status: open        # open | done
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

## 背景 / ゴール

（なぜやるか・何が達成されたら完了か）

## スコープ

（含むもの / 含まないもの）

## 配下 PBI

- [PBI-001](../pbis/PBI-001_login-screen.md)
```

## PBI のフォーマット

```markdown
---
id: PBI-001
title: <ユーザー視点の成果を一文で>
status: draft       # draft | ready | in-progress | done
epic: EPIC-001
requirements: ["REQ-F-001"]   # 要件定義書の対応要件
apis: ["API-003"]             # OpenSpec.yml の対応API（あれば）
screens: []                   # 画面仕様書の対応画面（あれば）
issues: []                    # 対応する GitHub Issue（例: "#12"）
branches: []                  # 作業ブランチ（例: "feature_#12"）
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

## 背景

（この PBI が EPIC のゴールにどう寄与するか）

## やること

（実装・作成するものの要約。詳細設計は仕様書側に委ね、ここでは書きすぎない）

## 受け入れ条件

- [ ] （検証可能な文で。例: ログイン成功時にトップ画面へ遷移する）
- [ ] （例: `make test-e2e` のログインシナリオが通る）

## 補足

（依存する PBI・技術的な前提・分割の経緯など）
```

## ステータス運用

| status | 意味 | 遷移のタイミング |
|---|---|---|
| draft | 起票のみ。内容が固まっていない | 起票直後 |
| ready | 受け入れ条件が確定し着手可能 | 内容確定時（Issue 化はこの時点以降） |
| in-progress | 作業ブランチで実装中 | ブランチ作成・着手時 |
| done | 受け入れ条件を満たしマージ済み | PR マージ時（`/harvest-knowledge` との併用を推奨） |

## 品質基準

- **ID 連鎖を切らさない**: PBI には requirements（対応する REQ-F-XXX）と、判明し次第 issues / branches を必ず記録する。この連鎖が将来の整合性監査（トレーサビリティ）の土台になる
- **重複起票しない**: ステップ1の調査で既存 PBI とのカバー関係を必ず確認する
- **索引と実体を一致させる**: ファイルの作成・更新と同じ回で必ず `docs/backlog/README.md` を更新する
- **受け入れ条件に曖昧語を使わない**: 「適切に」「正しく」ではなく、観測可能な結果・コマンドで書く
