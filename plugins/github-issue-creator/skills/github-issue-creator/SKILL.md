# /github-issue-creator

ユーザーの指示内容を整理し、GitHub Issueのドラフトを作成してGitHubにIssueを登録するスキルです。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0.5「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、ドラフトは報告のみで提示してそのまま作成まで実行する
- 曖昧な点（分割粒度・タイトル表現など）は合理的な仮定を置いて進め、採用した仮定を最終報告に明記する
- **例外**: 不可逆な破壊的操作・リスクの高い操作（Issueの削除、機密情報らしき内容を含む本文の投稿等）だけは、実行前に必ずユーザーに確認する（そもそも作成以外の書き込み操作は本スキルの対象外）

## 実行手段の優先順位（重要）

1. **GitHub MCP を優先する**: `create_issue` などGitHub MCPサーバーのツールが利用可能であれば、それを使う
2. **gh CLI で補完する**: 次の場合は `gh` コマンドを使う
   - GitHub MCPサーバーが未接続・認証エラーなどで利用できない場合
   - MCPツールが対応していない操作の場合（例: Issueテンプレートの取得、`gh api` でしか行えない操作）
3. どちらも利用できない場合は **ステップ0の前提チェックで検出し、作業を始める前に終了する**

## Usage

```
/github-issue-creator [Issueにしたい内容のテキスト | 元ネタとなるファイルのパス]
```

引数は省略可能です。省略した場合は、何をIssueにしたいかを実行前確認（ステップ0.5）で確認します。

## 全体フロー

### ステップ0: 前提チェック（最優先・作業前に必ず実行）

Issue内容の整理やドラフト作成を始める前に以下を確認し、**不足があればこの時点でユーザーに伝える**（後工程で気づくのは無駄なため）。

1. **GitHub MCPの接続確認**: `github` MCPサーバーのツール（`create_issue` など）が利用可能かを確認する
2. **gh CLIの確認**: `gh --version` でインストール有無、`gh auth status` で認証状態を確認する
3. 判定と通知
   - 両方利用可能 → そのまま続行（MCP優先）
   - 片方のみ利用可能 → どちらを使うかをユーザーに一言伝えて続行する
   - **どちらも利用できない → セットアップ手順（README参照: `/mcp` からのOAuth認証、または `gh auth login`）を案内して終了する。**ドラフト作成などの作業は行わない

### ステップ0.5: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。**引数・リポジトリの状態から自明な項目は聞かずに省略する**。全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **Issueにする内容**: `$ARGUMENTS` が空の場合のみ
2. **対象リポジトリ**: `git remote get-url origin` から一意に推定できる場合は聞かずにそれを採用して報告のみ。リモートがない・複数リモートで曖昧・別リポジトリの可能性がある場合のみ owner/repo を確認する
3. **分割粒度の方針**: 元ネタから多数の候補が見込まれる場合のみ（例: 機能単位か要件ID単位か）
4. **ラベル・アサイン・マイルストーンの付与方針**: 付与の要否が指示から読み取れない場合のみ

### ステップ1: 対象リポジトリの特定

1. カレントディレクトリで `git remote get-url origin` を確認し、対象リポジトリ（owner/repo）を特定する（曖昧な場合は実行前確認で確認済み）
2. 採用した owner/repo をドラフト報告（ステップ5）に明記する

### ステップ2: 入力の整理

1. `$ARGUMENTS` を確認する
   - パスらしき文字列の場合はまず存在確認し、存在すればそのファイル（要件定義書・仕様書・メモなど）を読み込んで元ネタとして扱う。存在しなければテキストとして扱う
   - テキストが渡された場合 → その内容をIssueの元ネタとして扱う
2. 元ネタから **Issue候補のリスト** を作る
   - 独立して対応できる作業単位ごとに1つのIssueに分割する
   - 要件定義書が元ネタの場合は、機能・要件ID（REQ-F-XXXなど）の単位で分割する
   - コード内のTODOコメントからの起票をユーザーが指示した場合は、Grepで `TODO|FIXME|HACK` を検索して候補を作る
3. 分割の粒度は実行前確認の回答（なければ「独立して対応できる最小単位」の原則）に従い、迷った箇所は仮定として報告に注記する

### ステップ3: リポジトリ規約の確認

1. 対象リポジトリのIssueテンプレートを確認する
   - ローカルにclone済みなら `.github/ISSUE_TEMPLATE/` ディレクトリをReadで確認する
   - ローカルにない場合は `gh api repos/{owner}/{repo}/contents/.github/ISSUE_TEMPLATE` で確認する（404なら未設定と判断）
   - gh CLIが利用できない場合はMCPのファイル取得ツール（`get_file_contents` など）で確認する。それも不可なら、既定テンプレートを使う旨をユーザーに伝えて続行する
2. リポジトリにテンプレートがあれば **その構成を優先** する。なければ本プラグインの既定テンプレート `${CLAUDE_PLUGIN_ROOT}/docs/issue-template.md` を使う
3. ラベルを付ける場合は、既存ラベル一覧（MCPのラベル取得ツール、または `gh label list`）を取得し、**存在するラベルのみ** 候補として提案する。新規ラベルの作成はユーザーが明示的に指示した場合のみ行う
4. アサイン先・マイルストーンを付ける場合も同様に、既存の一覧（MCPツール、または `gh api repos/{owner}/{repo}/milestones` / `gh api repos/{owner}/{repo}/assignees`）で存在を確認してから提案する

### ステップ4: ドラフト作成

1. Issue候補ごとにタイトル・本文のドラフトを作成する
   - タイトルは「何をするか」が一読でわかる簡潔な日本語（リポジトリの既存Issueが英語運用なら英語に合わせる）
   - 本文はステップ3で決めたテンプレート構成に沿って書く
2. コードベースの調査が本文の充実に役立つ場合（バグの該当箇所、再現手順、関連ファイル、影響範囲など）は、`issue-drafter` エージェントに調査を依頼する
   - 1回の依頼に1〜3件程度の関連するIssue候補をまとめて渡す
   - 返ってきた調査結果（該当ファイル・行、根拠、確度）を本文の「関連コード」「補足」セクションに反映する
   - 確度が「推測」の情報は、その旨を本文に明記して続行する
3. 機密情報（認証情報、内部URL、個人情報など）が本文に含まれていないか確認する。含まれていたら除去し、最終報告に明記する（除去できない形で本文に必要な場合のみ例外規定として確認する）

### ステップ5: ドラフトの報告

1. 提示前に、既存Issueとの重複を確認する（MCPの `search_issues`、または `gh issue list --search`）。既存Issueと明確に重複する候補は作成対象から外し、既存IssueのURLを添えて最終報告に記載する。重複か判断しきれない候補は作成対象に残し、本文に関連Issueとして既存URLを記載する
2. 全Issue候補を**報告として**一覧で提示し、ユーザーの応答を待たずにそのままステップ6へ進む
   - 対象リポジトリ（owner/repo）、番号、タイトル、本文の要約、付与予定のラベル・アサイン・マイルストーン
   - 分割粒度・タイトル表現などで仮定を置いた箇所はその旨を注記する

### ステップ6: Issue作成

1. 報告した候補を順に作成する。**ラベル・アサイン・マイルストーンは漏れなく指定する**
   - GitHub MCP: `create_issue`（owner, repo, title, body, labels, assignees, milestone を指定。ツールがmilestoneに対応しない場合はgh CLIで補完する）
   - gh CLIフォールバック: `gh issue create --repo <owner>/<repo> --title <title> --body-file <一時ファイル> --label <label> --assignee <user> --milestone <milestone>` （本文はエスケープ事故を避けるため一時ファイル経由で渡し、作成の成否にかかわらず削除する）
2. 作成に失敗した候補があっても残りの処理は継続し、失敗内容を記録する
3. レート制限や権限エラーが出た場合は、エラー内容と対処（PATのスコープ、`gh auth login` など）を案内する

### ステップ7: 結果サマリー

以下をユーザーに提示して終了する。

- 作成したIssueの一覧（番号・タイトル・URL）と対象リポジトリ
- 作成をスキップ（重複等）・失敗した候補とその理由
- **採用した仮定の一覧**（分割粒度・タイトル表現・ラベル選定などで仮定を置いた事項）
- 未解決の論点があれば次のアクション案（例: ラベル整備、マイルストーン設定）

## 制約

- ユーザーへの質問は実行前確認（ステップ0.5）の1回のみ。以降はドラフトを報告してそのまま作成まで完遂する（例外規定の破壊的・高リスク操作を除く）
- Issueの編集・クローズ・コメント追加・削除など、作成以外の書き込み操作は本スキルの対象外（ユーザーが求めた場合は対象外である旨を伝える）
- 機密情報・認証情報をIssue本文に含めない。トークンの値そのものを出力・ログに含めない
- 存在しないラベル・マイルストーン・アサイン先を指定しない（事前に存在を確認し、存在しないものは付与せず最終報告に記載する）
- 一括作成は1回の実行で最大10件までを目安とし、超える場合は10件ずつに分割して順次作成する（中断はしない）
