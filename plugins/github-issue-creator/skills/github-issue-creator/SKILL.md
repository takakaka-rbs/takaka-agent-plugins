# /github-issue-creator

ユーザーの指示内容を整理し、GitHub Issueのドラフトを作成して、承認を得たうえでGitHubにIssueを登録するスキルです。

## 実行手段の優先順位（重要）

1. **GitHub MCP を優先する**: `create_issue` などGitHub MCPサーバーのツールが利用可能であれば、それを使う
2. **gh CLI で補完する**: 次の場合は `gh` コマンドを使う
   - GitHub MCPサーバーが未接続・認証エラーなどで利用できない場合
   - MCPツールが対応していない操作の場合（例: Issueテンプレートの取得、`gh api` でしか行えない操作）
3. どちらも利用できない場合は、作成予定だったIssueの内容（タイトル・本文）をMarkdownで提示し、セットアップ手順（README参照）を案内して終了する

gh CLIを使う際は、実行前に `gh auth status` で認証状態を確認する。

## Usage

```
/github-issue-creator [Issueにしたい内容のテキスト | 元ネタとなるファイルのパス]
```

引数は省略可能です。省略した場合は、何をIssueにしたいかをユーザーに確認します。

## 全体フロー

### ステップ1: 対象リポジトリの特定

1. カレントディレクトリで `git remote get-url origin` を確認し、対象リポジトリ（owner/repo）を推定する
2. 推定した owner/repo を必ずユーザーに提示して確認する。リモートがない場合や別リポジトリに作成したい場合は、ユーザーに owner/repo を指定してもらう
3. **確認が取れるまでIssueを作成しない**

### ステップ2: 入力の整理

1. `$ARGUMENTS` を確認する
   - パスらしき文字列の場合はまず存在確認し、存在すればそのファイル（要件定義書・仕様書・メモなど）を読み込んで元ネタとして扱う。存在しなければテキストとして扱う
   - テキストが渡された場合 → その内容をIssueの元ネタとして扱う
   - 何も指定がない場合 → 「何をIssueにしますか？」とユーザーに確認する
2. 元ネタから **Issue候補のリスト** を作る
   - 独立して対応できる作業単位ごとに1つのIssueに分割する
   - 要件定義書が元ネタの場合は、機能・要件ID（REQ-F-XXXなど）の単位で分割を提案する
   - コード内のTODOコメントからの起票をユーザーが指示した場合は、Grepで `TODO|FIXME|HACK` を検索して候補を作る
3. 候補が複数になる場合は、分割の粒度をユーザーに確認する（細かすぎ・粗すぎの調整）

### ステップ3: リポジトリ規約の確認

1. 対象リポジトリのIssueテンプレートを確認する
   - ローカルにclone済みなら `.github/ISSUE_TEMPLATE/` ディレクトリをReadで確認する
   - ローカルにない場合は `gh api repos/{owner}/{repo}/contents/.github/ISSUE_TEMPLATE` で確認する（404なら未設定と判断）
   - gh CLIが利用できない場合はMCPのファイル取得ツール（`get_file_contents` など）で確認する。それも不可なら、既定テンプレートを使う旨をユーザーに伝えて続行する
2. リポジトリにテンプレートがあれば **その構成を優先** する。なければ本プラグインの既定テンプレート `${CLAUDE_PLUGIN_ROOT}/docs/issue-template.md` を使う
3. ラベルを付ける場合は、既存ラベル一覧（MCPのラベル取得ツール、または `gh label list`）を取得し、**存在するラベルのみ** 候補として提案する。新規ラベルの作成はユーザーが明示的に指示した場合のみ行う
4. アサイン先・マイルストーンを付ける場合も同様に、既存の一覧（MCPツール、または `gh api repos/{owner}/{repo}/milestones` / `gh api repos/{owner}/{repo}/assignees`）で存在を確認してから提案する

### ステップ4: ドラフト作成

1. Issue候補ごとにタイトル・本文のドラフトを作成する
   - タイトルは「何をするか」が一読でわかる簡潔な日本語（リポジトリの既存Issueが英語運用なら英語に合わせる）
   - 本文はステップ3で決めたテンプレート構成に沿って書く
2. コードベースの調査が本文の充実に役立つ場合（バグの該当箇所、再現手順、関連ファイル、影響範囲など）は、`issue-drafter` エージェントに調査を依頼する
   - 1回の依頼に1〜3件程度の関連するIssue候補をまとめて渡す
   - 返ってきた調査結果（該当ファイル・行、根拠、確度）を本文の「関連コード」「補足」セクションに反映する
   - 確度が「推測」の情報はその旨を本文に明記するか、ユーザーに確認する
3. 機密情報（認証情報、内部URL、個人情報など）が本文に含まれていないか確認する。含まれていたら除去し、ユーザーに知らせる

### ステップ5: ユーザー承認（必須）

1. 提示前に、既存Issueとの重複を確認する（MCPの `search_issues`、または `gh issue list --search`）。重複の可能性がある候補には既存IssueのURLを添えて注記する
2. 全Issue候補を一覧で提示する
   - 番号、タイトル、本文の要約（または全文）、付与予定のラベル・アサイン・マイルストーン
3. ユーザーに以下を確認する
   - どの候補を作成するか（全部 / 選択 / 破棄）
   - タイトル・本文・ラベルの修正有無
4. 修正指示があればドラフトを更新し、再提示する
5. **ユーザーの明示的な承認なしにIssueを作成しない。** 「たぶんOKだろう」で進めない

### ステップ6: Issue作成

1. 承認された候補を順に作成する。**承認されたラベル・アサイン・マイルストーンは漏れなく指定する**
   - GitHub MCP: `create_issue`（owner, repo, title, body, labels, assignees, milestone を指定。ツールがmilestoneに対応しない場合はgh CLIで補完する）
   - gh CLIフォールバック: `gh issue create --repo <owner>/<repo> --title <title> --body-file <一時ファイル> --label <label> --assignee <user> --milestone <milestone>` （本文はエスケープ事故を避けるため一時ファイル経由で渡し、作成の成否にかかわらず削除する）
2. 作成に失敗した候補があっても残りの処理は継続し、失敗内容を記録する
3. レート制限や権限エラーが出た場合は、エラー内容と対処（PATのスコープ、`gh auth login` など）を案内する

### ステップ7: 結果サマリー

以下をユーザーに提示して終了する。

- 作成したIssueの一覧（番号・タイトル・URL）
- 作成をスキップ・失敗した候補とその理由
- 未解決の論点があれば次のアクション案（例: ラベル整備、マイルストーン設定）

## 制約

- **ユーザーの承認前にIssueを作成しない**（Dry-run的にドラフト提示までで止める）
- Issueの編集・クローズ・コメント追加など、作成以外の書き込み操作は本スキルの対象外（ユーザーが求めた場合は対象外である旨を伝える）
- 機密情報・認証情報をIssue本文に含めない。トークンの値そのものを出力・ログに含めない
- 存在しないラベル・マイルストーン・アサイン先を指定しない（事前に存在を確認する）
- 対象リポジトリの確認なしに `git remote` の推定だけで作成しない
- 一括作成は1回の実行で最大10件までを目安とし、超える場合は分割実行を提案する
