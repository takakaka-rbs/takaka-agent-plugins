# screen-specificater

要件定義書（REQ-F-XXX）とOpenAPI仕様書（API-XXX）を参照しながら、画面仕様書（Markdown）を作成・更新するプラグインです。

スクリーンショットにコンポーネント名を添えただけの画面カタログではなく、**実装者が仕様書だけで画面を正しく実装できる「画面の機能仕様書」** として通用するレベルまで記述することを品質基準としています。

## 概要

ユーザーが要件定義書や画面のメモを渡すと、`/screen-specificater` スキルが以下を行います。

1. **実行前確認（唯一の質問タイミング）**: 新規/更新・出力先・参照文書の場所・注釈付き画像の提供予定など方針判断が必要な事項を、最大4問まで最初に1回だけ確認する（文脈から自明な項目は聞かない）
2. 要件定義書の機能要件（REQ-F-XXX）から必要な画面の一覧・画面遷移図（Mermaid）を設計して報告する（承認は待たない）
3. 共通規約（権限による表示制御・エラー表示・ローディング・バリデーションのタイミング・対象デバイス）を既存コードベースの慣習・一般的なUXの慣習から決定する
4. 不足している情報を「コードベース調査」「Web調査」「合理的な仮定（注記 + `OPEN-XXX` 記録）」で埋める
5. 各画面をチェックリスト（コンポーネント配置・イベントハンドラー仕様・表示制御・状態仕様）に沿って記述する
6. 要件・APIとのトレーサビリティ（REQ-F-XXX / API-XXX 参照）と画面ID（SCR-XXX）を整備し、採用した仮定と未確定事項を最終報告で開示する

## 文書の役割分担

```
要件定義書（requirements-specificater） … サービスが満たすべき要件を定義する
  ↑ REQ-F-XXX を参照
OpenSpec.yml（openapi-specificater）    … 要件を満たす具体的なAPIを定義する
  ↑ API-XXX を参照
画面仕様書（本プラグイン）              … 要件を満たす具体的な画面を定義する
```

参照は常に「画面仕様書 → 要件定義書」「画面仕様書 → OpenSpec.yml」の一方向です。API呼び出しを伴うイベントには対応する API-XXX を必ず記載し、参照先の文書に画面番号は書きません。対応する要件・APIが存在しない場合は、先に `/requirements-specificater`・`/openapi-specificater` で追加します。

## 成果物のファイル構成

画面仕様書は「全体目次1ファイル + 画面ごとに1ファイル」で構成されます（出力先は実行前確認で決定します）。

```
docs/screens/
├── index.md            # 画面一覧・画面遷移図（Mermaid）・共通規約
├── SCR-001-login.md    # 画面ごとの詳細仕様
├── SCR-002-user-list.md
└── images/             # 注釈付き画像
```

### コンポーネント配置は「注釈付き画像 + 対応表」で表現する

レイアウトは、コンポーネントごとの番号を矢印などで注釈した画像（Figmaのエクスポート、Playwrightのスクリーンショットへの注釈など。**ユーザーが用意**）で表現します。状態変化の前後など、1画面に複数枚あってかまいません。

画像の注釈番号は画像ごとに振り直されるため、仕様の記述には使いません。画像ごとに「注釈番号 → CMP-XXX」対応表を置き、イベントハンドラー・表示制御・状態の仕様はすべて一意なID（CMP-XXX / EVT-XXX）側に記述します。画像が未準備でも、プレースホルダー + OPEN-XXX 記録で仕様の記述を先行できます。

## 記述レベルの基準

見た目のスクリーンショットを貼っただけの文書は成果物として不合格です。各画面について以下が仕様書から読み取れることを完成条件とします。

- 誰が何のためにこの画面を使うのか（利用場面・対応要件）
- どのコンポーネントがどこに配置されているか（注釈付き画像 + CMP対応表）
- 各イベントハンドラーの機能（トリガー → 事前条件 → 処理 → 成功時 → エラーごとの挙動。API呼び出しは API-XXX 参照）
- 権限・条件による表示の出し分け
- 初期表示・ローディング・空・エラーの各状態の表示

詳細な合否基準と良い例・悪い例は [docs/screen-spec-authoring-guide.md](docs/screen-spec-authoring-guide.md) を参照してください。

## 構成要素

| コンポーネント | ファイル | 説明 |
|---|---|---|
| Skill | `skills/screen-specificater/SKILL.md` | `/screen-specificater` スラッシュコマンド。実行前確認から出力までの全体フローを制御する |
| Agent | `agents/screen-researcher.md` | コードベース調査（ルーティング・コンポーネント・イベントハンドラー・Storybook等）・Web調査を担当するサブエージェント |
| テンプレート | `docs/screen-spec-template.md` | index.md と画面別ファイルのベーステンプレート（記入ガイド付き） |
| 記述レベルガイド | `docs/screen-spec-authoring-guide.md` | 「機能仕様書レベル」の合否基準チェックリストと良い例・悪い例 |

## ID採番ルール

| ID | 対象 | 説明 |
|---|---|---|
| SCR-XXX | 画面 | 画面の不変ID。ファイル名・遷移図・他画面や実装・テストからの参照キーになる |
| CMP-XXX | コンポーネント | 画面仕様書全体で一意。画像の注釈番号とは対応表で紐付ける |
| EVT-XXX | イベントハンドラー | 画面仕様書全体で一意。CMP-XXXに紐付く |
| OPEN-XXX | 未確定事項 | 本書スコープの未確定事項（要件定義書・OpenSpec.ymlのOPEN-XXXとは独立した採番） |

`XXX` は `001` から始まる3桁の連番です。**既存の仕様書を更新する際、既存の SCR-XXX・CMP-XXX・EVT-XXX は変更しません**（実装・テスト・他文書からの参照が壊れるため）。新規追加には新しい連番を割り当てます。

## 使い方

```
/screen-specificater
```

新規作成・既存ファイルの更新・要件定義書とOpenSpec.ymlの場所・出力先パスなどは、実行前確認（最大4問・自明な項目は省略）で最初に1回だけ確認し、以降は質問せずに完遂します。

既存の画面仕様書を更新する場合は index.md のパスを引数に渡せます。

```
/screen-specificater docs/screens/index.md
```

要件定義書のパスや画面のメモをそのまま渡して開始することもできます。

```
/screen-specificater docs/requirements/my-service.md
/screen-specificater 在庫管理システムの画面。商品一覧・商品詳細・在庫調整の3画面が必要。
```

## インストール

マーケットプレイス経由でインストールします（テンプレート・記述レベルガイドを含むプラグイン全体が配置され、`${CLAUDE_PLUGIN_ROOT}` が解決されます）。

```bash
# マーケットプレイス未登録の場合のみ
claude plugin marketplace add /path/to/takaka-agent-plugins

claude plugin install screen-specificater@takaka-agent-plugins
```
