# /screen-specificater

画面仕様書（Markdown形式）を、ユーザーとの対話・コードベース調査・Web調査を通じて作成・更新するスキルです。

## この文書が扱う抽象度（重要）

画面仕様書は「要件を満たす具体的な画面」を定義する文書です。

- 書くこと: 画面一覧・画面遷移に加えて、各画面のコンポーネント配置（注釈付き画像 + 対応表）・イベントハンドラーの機能仕様（トリガー → 処理 → 結果・エラー時挙動）・表示制御（権限・条件による出し分け）・状態仕様（初期表示・ローディング・空・エラー）。**実装者が本書だけで画面を正しく実装できるレベル**まで書く
- 書かないこと: APIのリクエスト/レスポンス定義（OpenSpec.ymlの担当。`API-XXX` で参照する）、色・余白・フォントなどビジュアルデザインの詳細（デザインデータの担当）、フレームワーク・状態管理ライブラリなどの実装方式
- 参照方向は常に「本書 → 要件定義書」「本書 → OpenSpec.yml」の一方向。各画面の対応要件に要件ID（REQ-F-XXX）を、API呼び出しを伴うイベントに API ID（API-XXX）を記載し、参照先の文書に画面IDは書かない

見た目のスクリーンショットを貼っただけの文書や、コンポーネント名の羅列だけの文書は本スキルの成果物として不合格です。記述レベルの合否基準は記述レベルガイドに従います。

## Usage

```
/screen-specificater [既存の画面仕様書（index.md）のパス | 要件定義書のパス | 画面のメモ・ドラフトのテキスト]
```

引数は省略可能です。省略した場合は、新規作成か既存ファイルの更新かをユーザーに確認します。

## ベースドキュメント

- テンプレート: `${CLAUDE_PLUGIN_ROOT}/docs/screen-spec-template.md`（index.md と画面別ファイルの両方の雛形を含む）
- 記述レベルガイド: `${CLAUDE_PLUGIN_ROOT}/docs/screen-spec-authoring-guide.md`

テンプレートには文書の位置づけ・ID採番ルール・各セクションの記入ガイド（`<!-- GUIDE: -->` コメント）が、記述レベルガイドには「不足/十分」の具体例と完成判定チェックリストが含まれています。**作業前に必ず両方を読み込んでください。**

## ファイル構成

画面仕様書は「全体目次1ファイル + 画面ごとに1ファイル」で構成します。

```
docs/screens/
├── index.md            # 画面一覧・画面遷移図（Mermaid）・共通規約
├── SCR-001-login.md    # 画面ごとの詳細仕様（ファイル名は SCR-XXX-<画面名スラッグ>.md）
├── SCR-002-user-list.md
└── images/             # 注釈付き画像（SCR-XXX-<用途>.png など）
```

出力先のルート（上記例の `docs/screens/`）はユーザーに確認します。

## ID採番ルール

| ID | 対象 | 説明 |
|---|---|---|
| SCR-XXX | 画面 | 画面の不変ID。ファイル名・遷移図・他画面からの参照に使う |
| CMP-XXX | コンポーネント | **画面仕様書全体で一意**。画像の注釈番号は画像ごとに振り直されるため、IDとしては使わない |
| EVT-XXX | イベントハンドラー | **画面仕様書全体で一意**。CMP-XXXに紐付く |
| OPEN-XXX | 未確定事項 | 本書スコープの未確定事項（要件定義書・OpenSpec.ymlのOPEN-XXXとは独立した採番） |

`XXX` は `001` から始まる3桁の連番です。**既存の仕様書を更新する場合、既存の SCR-XXX・CMP-XXX・EVT-XXX は変更・再採番しない**（実装・テスト・他文書からの参照を壊さないため）。新規追加には欠番を避けて新しい連番を割り当てます。

## 画像（コンポーネント配置）の扱い

- レイアウトは、コンポーネントごとの番号（No.1, No.2, …）を矢印などで注釈した画像で表現する。画像はユーザーが用意する（Figmaのエクスポート、Playwrightのスクリーンショットへの注釈など）。スキルは画像を生成・加工しない
- 1画面に画像が複数あってよい（状態変化の前後、モーダル展開時など）。画像ごとに「注釈番号 → CMP-XXX」の対応表を必ず置き、仕様の記述はすべて CMP-XXX / EVT-XXX 側に書く
- 画像が未準備の段階では、画像枠をプレースホルダー（`_(画像未添付)_`）のまま残して `OPEN-XXX` に記録し、コンポーネント一覧表・イベント仕様の記述は画像なしで先に進めてよい

## 全体フロー

### ステップ1: 状況確認

1. `$ARGUMENTS` を確認する
   - 既存の画面仕様書（index.md）のパスが指定された場合 → indexと画面別ファイルを読み込み、更新作業として続きから進める
   - 要件定義書（Markdown）のパスが指定された場合 → 新規作成として扱い、それをインプットに使う
   - 画面のメモ・テキストが渡された場合 → 新規作成として扱い、その内容を入力情報とする
   - 何も指定がない場合 → 「新規作成」か「既存ファイルの更新」かをユーザーに確認する
2. 要件定義書の場所を確認する。指定がなければ `docs/requirements/` 配下などを探し、見つからない場合はユーザーに尋ねる
   - 要件定義書が存在しない場合はその旨を伝え、先に `/requirements-specificater` での作成を推奨する。ユーザーが「要件定義書なしで進める」ことを選んだ場合のみ、対応要件を空にせず `OPEN-XXX` に「要件定義書との対応付けが未実施」と記録して進める
3. OpenSpec.yml の場所を確認する。指定がなければ `docs/api/` 配下などを探す。存在しない場合、API呼び出しを伴うイベントの API-XXX 参照は `OPEN-XXX` に「OpenSpec.ymlとの対応付けが未実施」と記録して進める（`/openapi-specificater` での作成を案内する）
4. 出力先ルートをユーザーに確認する。**毎回必ず確認し、固定パスを決め打ちしない**
   - 例として `docs/screens/` のような構成を提案してもよいが、最終的にはユーザーの指定に従う
5. 新規作成の場合はテンプレートをコピーして作業ファイルとする。既存ファイルの場合は読み込み、`OPEN-XXX`・`_(未記入)_`・`_(画像未添付)_` 箇所・記述レベルガイドのチェックリスト違反を洗い出して状況を把握する

### ステップ2: 画面一覧・遷移の設計・合意

詳細を書き始める前に、まず「どんな画面があるか」の骨組みをユーザーと合意する。

1. 要件定義書の機能要件（REQ-F-XXX）・ユースケース（UC-XXX）を読み、必要な画面の候補を洗い出す
2. 次の形式の画面一覧表を提示し、過不足をユーザーに確認する

   | SCR ID | 画面名 | 概要 | 対応要件 |
   |---|---|---|---|
   | SCR-001 | ログイン | 認証を行い、成功時にダッシュボードへ遷移する | REQ-F-001 |

3. 画面遷移図（Mermaid）を作成し、遷移条件（認証状態・権限・操作）もあわせて確認する
4. この時点で以下の共通規約もあわせて確認・決定する（既存コードベースがあれば `screen-researcher` に現行の慣習を調査させ、それに合わせることを提案する）
   - 認証・権限による表示制御の基本方針 / エラー表示の方式（トースト・インライン・ダイアログの使い分け）/ ローディング表示の方式 / バリデーションのタイミング（入力時・送信時）/ 対象デバイス・ブレークポイント
5. 合意した一覧・遷移図・共通規約を index.md に反映し、画面別ファイルの骨組み（SCR-XXX-<スラッグ>.md）を作成する

### ステップ3: 不足情報の調査・質問

各画面の詳細を書くために足りない情報を3種類に分類して埋める。

1. **(a) コードベース調査で埋まる可能性がある項目**: 既存実装のルーティング・画面コンポーネント・イベントハンドラー・バリデーション・Storybookのストーリー、現行画面との互換性制約など → `screen-researcher` エージェントに依頼する
2. **(b) Web調査で埋まる可能性がある項目**: UIパターンの一般的な慣習（フォームバリデーションのタイミング、無限スクロールとページネーションの使い分け、エラー表示のUXなど）→ `screen-researcher` エージェントに依頼する
3. **(c) ユーザーに確認するしかない項目**: 業務フロー上の画面の使われ方、権限ごとの見せ方の線引き、エラー時に業務としてどうさせたいか、注釈付き画像の提供など → ユーザーに質問する
   - 一度に大量の質問をしない。画面単位またはテーマ単位で3〜5問程度ずつ確認する
   - 「わからない」「後で決める」という回答は無理に埋めず、`OPEN-XXX` として記録する

### ステップ4: 画面ごとの詳細記述

画面一覧の各行について、記述レベルガイドの基準で詳細を書く。

1. 注釈付き画像の提供をユーザーに依頼し、`images/` 配下への配置とファイル名を確認する。未準備なら「画像の扱い」の運用（プレースホルダー + OPEN-XXX）で先に進める
2. 画像ごとに「注釈番号 → CMP-XXX」対応表を作り、コンポーネント一覧表（CMP-XXX / 名称 / 種別 / 概要）を記述する
3. 各イベントハンドラー（EVT-XXX）を「トリガー → 事前条件 → 処理 → 成功時の結果 → エラー時の挙動」で記述する。API呼び出しを伴う場合は必ず API-XXX を参照する（OpenSpec.ymlに該当APIがない場合は `/openapi-specificater` での追加を案内し、`OPEN-XXX` に記録する）
4. 表示制御（権限・条件による出し分け）と状態仕様（初期表示・ローディング・空・エラー）を記述する
5. 1画面を書き終えるごとに、記述レベルガイドの「画面ごと」チェックリストで自己監査する。未達の項目を残したまま次へ進まない（情報が得られない場合は `OPEN-XXX` に記録した上で進む）

### ステップ5: ID採番・整合性チェック

1. SCR-XXX・CMP-XXX・EVT-XXX が全対象に一意に振られていることを確認する。**既存IDは変更・再採番しない**
2. 対応要件の全IDが要件定義書に実在すること、参照する API-XXX が OpenSpec.yml に実在することを突き合わせて確認する。存在しないIDは書かない
3. 逆方向のカバレッジを確認する: 要件定義書の画面関連の機能要件のうち、どの画面からも参照されていないものを洗い出し、「画面追加が必要」か「対象外」かをユーザーに確認する
4. index.md の画面一覧・遷移図と画面別ファイルの実体が一致していることを確認する（一覧にあるのにファイルがない・逆がない）

### ステップ6: 検証・出力・サマリー

1. テンプレート由来の `<!-- GUIDE: -->` コメントと、未使用のテンプレート雛形（`_(未記入)_` のプレースホルダー行）を成果物から削除する（`_(画像未添付)_` は OPEN-XXX が残っている間は残してよい）
2. 埋め込んだ画像パス・文書間リンクの参照先がすべて存在すること、Mermaidの遷移図が構文として正しいことを確認する
3. 出力先に保存する
4. ユーザーに以下をサマリーとして提示する
   - 画面一覧（SCR-XXX / 画面名 / 対応要件）
   - 要件カバレッジ（要件定義書のREQ-F-XXXのうち対応済み・未対応のもの）
   - API参照の一覧（EVT-XXX → API-XXX）と、OpenSpec.yml側に存在しなかったもの
   - 未確定事項（OPEN-XXX）の一覧と、解決のために誰に何を確認すべきか（画像未提供の画面を含む）
   - 記述レベルチェックリストの充足状況（未達があれば正直に列挙する）

## 中断・再開

- 対話は複数回のセッションに分けてよい
- 再実行時は index.md と画面別ファイルを読み込み、`OPEN-XXX`・`_(未記入)_`・`_(画像未添付)_` 箇所、チェックリスト未達の画面から再開する
- 1回のセッションで全画面を書ききる必要はない。ユーザーが切り上げたい場合は、その時点の状態で保存し、残課題を `OPEN-XXX` にまとめて終了する

## 制約

- APIのリクエスト/レスポンス定義・テーブル設計・実装方式を本書に書かない（APIは API-XXX 参照に留める）
- 要件定義書に存在しない要件ID（REQ-F-XXX）、OpenSpec.ymlに存在しないAPI ID（API-XXX）を発明しない。参照先文書自体の変更は行わず、`/requirements-specificater`・`/openapi-specificater` に委ねる
- 既存の SCR-XXX・CMP-XXX・EVT-XXX を、ユーザーの明示的な合意なく変更・再採番しない（破壊的変更）
- 画像はユーザーが用意する。スキルが画像の内容（注釈番号と実際の配置の対応）を推測で確定させない。読み取れない・不整合がある場合はユーザーに確認する
- 権限の線引き・エラー時の業務的な挙動などユーザーが決めるべき事項を、調査結果や推測だけで勝手に確定させない
- 機密情報・認証情報・未公開のスクリーンショットをWeb調査に渡さない（`screen-researcher` エージェントへの依頼内容にも含めない）
- 記述レベルガイドのチェックリストを満たさない状態を「完成」と報告しない（未達は必ずサマリーで開示する）
- 出力先パスは必ずユーザーに確認してから書き込む
