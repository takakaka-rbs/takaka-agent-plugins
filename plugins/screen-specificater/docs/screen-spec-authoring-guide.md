# 画面仕様書 記述レベルガイド

screen-specificater プラグインが生成する画面仕様書の品質基準です。
目標は **「実装者が本書だけで画面を正しく実装できる機能仕様書」** であり、
「スクリーンショットにコンポーネント名を添えただけの画面カタログ」ではありません。

`/screen-specificater` スキルは、各画面を書き終えるたびに本ガイドのチェックリストで
自己監査し、満たさない項目を残したまま完成扱いにしてはいけません。

## 1. 何が「不足」で何が「十分」か

次の例は体裁としては画面仕様書に見えますが、機能仕様書としては不足しています。

```markdown
<!-- ✗ 不足している例 -->
### EVT-001 ログインボタン押下

| 項目 | 内容 |
|---|---|
| 対象コンポーネント | CMP-003 |
| トリガー | クリック |
| 処理 | ログインする |
| 成功時 | 次の画面へ遷移 |
| エラー時 | エラーを表示 |
```

この記述では以下の質問に答えられません。**読者がこれらを設計者に口頭で確認しなければ
ならない時点で、仕様書として機能していません。**

- 「ログインする」とは具体的にどのAPIを呼ぶことか（OpenSpec.ymlのどのAPI-XXXか）
- 押下前にバリデーションは走るのか。未入力のまま押せるのか
- 「次の画面」とはどの画面（SCR-XXX）か
- 認証失敗（401）と通信エラー（5xx）で挙動は同じなのか
- エラー表示はどこに出るのか（トースト・インライン・ダイアログ）。入力値は消えるのか
- 処理中にもう一度押せるのか（多重送信）

同じイベントを本プラグインの基準で書くと次のようになります。

```markdown
<!-- ○ 十分な例 -->
### EVT-001 ログイン実行

| 項目 | 内容 |
|---|---|
| 対象コンポーネント | CMP-003 ログインボタン |
| トリガー | クリック（処理中は共通規約4.3に従いボタンを無効化し、多重送信を防ぐ） |
| 事前条件 | CMP-001・CMP-002 のバリデーション（本書 4章）を全項目通過していること。未通過の場合はAPIを呼ばず、該当項目にインラインでエラーを表示する |
| 処理 | API-001（POST /auth/login）を CMP-001 のメールアドレス・CMP-002 のパスワードで呼び出す |
| 成功時 | 取得したセッションを保存し、SCR-002 ダッシュボードへ遷移する |
| エラー時 | 401: フォーム上部にインラインで「メールアドレスまたはパスワードが正しくありません」を表示し、画面に留まる（入力値は保持、パスワードのみクリア）。<br>429: 「試行回数が上限に達しました。しばらくしてからお試しください」を表示し、CMP-003 を60秒間無効化する。<br>5xx・通信エラー: 共通規約4.2のとおりトーストで通知し、画面に留まる（入力値は保持） |
```

## 2. チェックリスト

### 2.1 文書全体（index.md）

- [ ] 参照文書（要件定義書・OpenSpec.yml）への実在するパスがある
- [ ] 画面一覧の全行に SCR-XXX・画面別ファイルへのリンク・対応要件（REQ-F-XXX）がある
- [ ] 画面一覧の全画面が遷移図に登場し、遷移図の全画面が一覧にある（孤立画面・幽霊画面がない）
- [ ] 条件付きの遷移（認証状態・権限・操作結果による分岐）に条件がラベルとして書かれている
- [ ] 共通規約に以下がある: 認証・権限による表示制御の基本方針 / エラー表示の方式 /
      ローディング表示の方式 / バリデーションのタイミング / 対象デバイス・ブレークポイント

### 2.2 各画面（SCR-XXX-*.md）ごと

- [ ] URL/ルート・対応要件（REQ-F-XXX）・対象ユーザー/権限・遷移元/遷移先がある
- [ ] 画面の目的・利用場面が「誰が何のために使うか」で書かれている
- [ ] レイアウト画像がある（未提供の場合は `_(画像未添付)_` と OPEN-XXX が記録されている）
- [ ] 各画像に「注釈番号 → CMP-XXX」対応表があり、注釈番号の全番号が表に載っている
- [ ] コンポーネント一覧に画面上の全コンポーネントがあり、操作可能なものには関連イベント（EVT-XXX）がある
- [ ] 入力系コンポーネント全項目に制約とエラーメッセージがある（入力がない画面は「なし」と明記）
- [ ] 表示制御（画面固有の出し分け）が書かれている（なければ「なし」と明記）
- [ ] 状態仕様に初期表示・ローディング・空（0件）・エラーの4状態がある
      （一覧・詳細など データ取得を伴う画面は特に。該当しない状態は「なし」と明記）
- [ ] 初期表示でデータ取得する場合、取得元の API-XXX が書かれている

### 2.3 各イベントハンドラー（EVT-XXX）ごと

- [ ] 対象コンポーネント（CMP-XXX）とトリガーが特定されている
- [ ] 事前条件がある（なければ「なし」と明記。バリデーション未通過時の挙動を含む）
- [ ] 処理が具体的である。API呼び出しを伴う場合は OpenSpec.yml に実在する API-XXX を参照している
- [ ] 成功時の結果が具体的である（遷移先は SCR-XXX、表示更新は対象と内容まで）
- [ ] エラー時の挙動が「起こりうるエラーごと」に書かれている
      （参照する API-XXX が定義するエラーステータスを網羅し、表示方法・遷移の有無・入力値の保持を含む）
- [ ] 多重実行の扱いが読み取れる（共通規約に従う場合はその旨、固有の挙動なら明記）

### 2.4 書いてはいけないもの

- [ ] APIのリクエスト/レスポンスのフィールド定義を書いていない（API-XXX 参照に留めている）
- [ ] 色・余白・フォントサイズなどビジュアルデザインの詳細を書いていない
- [ ] 特定のフレームワーク・UIライブラリ・状態管理の実装方式を書いていない
      （コンポーネント種別は TextInput / Button / Table のような抽象名にする）

## 3. ID採番・トレーサビリティ

| 項目 | ルール |
|---|---|
| SCR-XXX / CMP-XXX / EVT-XXX | 001からの3桁連番（必要なら桁を増やす）。CMP・EVT は**画面仕様書全体で一意**。**既存IDは変更・再採番しない** |
| 画像の注釈番号 | 画像ごとに振り直してよい。IDではないため、必ず対応表で CMP-XXX に変換する |
| 対応要件 | 要件定義書に実在する `REQ-F-XXX` のみを書く。存在しないIDを発明しない |
| API参照 | OpenSpec.yml に実在する `API-XXX` のみを書く。存在しないIDを発明しない |
| OPEN-XXX | 本書スコープの未確定事項を管理（要件定義書・OpenSpec.ymlのOPEN-XXXとは独立）。採番は index + 全画面ファイルを通して一意にする |
| 参照方向 | 常に「本書 → 要件定義書」「本書 → OpenSpec.yml」の一方向。参照先に画面番号を書かせない |

対応する要件が要件定義書に存在しない画面を追加したくなった場合は先に
requirements-specificater で要件を、参照したいAPIが OpenSpec.yml に存在しない場合は先に
openapi-specificater でAPIを追加してもらうようユーザーに案内します。

## 4. 記述スタイル

- 説明文は日本語で書く。URL・エラーコード・コンポーネント種別は英語でよい
- 挙動は「〜する」と断定形で書く（「〜が望ましい」「〜を想定」のような曖昧表現にしない。
  未確定なら OPEN-XXX に回す）
- 画面・API・要件への言及は必ず ID（SCR-XXX / API-XXX / REQ-F-XXX）を添える。
  「一覧画面」「ログインAPI」のような名前だけの参照にしない
- エラーメッセージ・文言は実際に表示する文字列をそのまま書く（「エラーメッセージを表示」で済ませない）
- 共通規約（index.md 4章）でカバーされる挙動は画面別ファイルに繰り返さない。
  共通規約と異なる場合のみ、その画面で明示的に上書きする
