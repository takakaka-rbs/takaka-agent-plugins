# /openapi-specificater

OpenAPI 3.1 形式のAPI仕様書（OpenSpec.yml）を、コードベース調査・Web調査を通じて作成・更新するスキルです。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、API一覧などの設計も報告のみで最後まで実行する
- 作業中に判明した不明点は、調査で埋まらなければ合理的な仮定を置いて進める。仮定した箇所は `description` 等に「（仮定）」と注記して `x-open-issues` にも記録し、採用した仮定の一覧を最終報告に明記する
- **例外**: 不可逆な破壊的操作（既存ファイルの削除、公開済みAPIの `x-api-id`・`operationId`・パス/フィールド名の変更等の破壊的変更）だけは、実行前に必ずユーザーに確認する

## この文書が扱う抽象度（重要）

OpenSpec.yml は「要件を満たす具体的なAPI」を定義する文書です。

- 書くこと: エンドポイント・スキーマの定義に加えて、各APIの目的・利用場面・事前条件・ビジネスルール・副作用・エラー発生条件・具体例。**実装者・利用者が本書だけでAPIを正しく実装・利用できるレベル**まで書く
- 書かないこと: 画面レイアウト・画面遷移（画面仕様書の担当）、サーバー内部の実装方式・テーブル設計
- 参照方向は常に「本書 → 要件定義書」の一方向。各APIの `x-requirements` に要件ID（REQ-F-XXX）を記載し、要件定義書側にAPI番号は書かない

構文的に正しいだけの最低限のYAML（summaryと型定義だけのもの）は本スキルの成果物として不合格です。記述レベルの合否基準は記述レベルガイドに従います。

## Usage

```
/openapi-specificater [既存のOpenSpec.ymlのパス | 要件定義書のパス | APIのメモ・ドラフトのテキスト]
```

引数は省略可能です。省略した場合は、新規作成か既存ファイルの更新かを実行前確認（ステップ0）で確認します。

## ベースドキュメント

- テンプレート: `${CLAUDE_PLUGIN_ROOT}/docs/openapi-template.yml`
- 記述レベルガイド: `${CLAUDE_PLUGIN_ROOT}/docs/openapi-authoring-guide.md`

テンプレートには文書の位置づけ・ID採番ルール・各セクションの記入ガイド（`# GUIDE:` コメント）が、記述レベルガイドには「不足/十分」の具体例と完成判定チェックリストが含まれています。**作業前に必ず両方を読み込んでください。**

## 全体フロー

### ステップ0: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。**引数・プロジェクトの文脈から自明な項目は聞かずに省略する**。全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **新規作成か既存更新か・入力元**: `$ARGUMENTS` から判別できない場合のみ
2. **出力先パス**: 引数で既存ファイルが指定された場合はそのパスを使い、聞かない。新規作成で慣習的な場所（`docs/api/` 等）が既にプロジェクトに存在する場合はそれを採用して報告のみ。どちらでもない場合に確認する
3. **要件定義書の扱い**: `docs/requirements/` 配下などを探しても見つからない場合のみ、「場所の指定」または「要件定義書なしで進める」ことを確認する（なしで進める場合は `x-open-issues` に「要件定義書との対応付けが未実施」と記録する）
4. **共通規約の方針**: 既存コードベースの調査で現行の慣習（認証方式・エラー形式・命名規則等）が判明する場合は聞かずにそれへ合わせる。既存の慣習がなく、入力からも判断できない場合のみ確認する

### ステップ1: 状況確認

1. `$ARGUMENTS` を確認する
   - 既存のOpenSpec.yml（YAMLファイル）のパスが指定された場合 → そのファイルを読み込み、更新作業として続きから進める
   - 要件定義書（Markdown）のパスが指定された場合 → 新規作成として扱い、それをインプットに使う
   - APIのメモ・テキストが渡された場合 → 新規作成として扱い、その内容を入力情報とする
2. 要件定義書の場所を確認する。指定がなければ `docs/requirements/` 配下などを探す（見つからない場合の扱いは実行前確認で決定済み）
3. 新規作成の場合はテンプレートをコピーして作業ファイルとする。既存ファイルの場合は読み込み、`x-open-issues`・`_(未記入)_` 箇所・記述レベルガイドのチェックリスト違反を洗い出して状況を把握する

### ステップ2: API一覧の設計・報告

詳細を書き始める前に、まず「どんなAPIを作るか」の骨組みを固めて報告する（承認は待たない）。

1. 要件定義書の機能要件（REQ-F-XXX）・ユースケース（UC-XXX）を読み、必要なリソースとエンドポイントの候補を洗い出す
2. 次の形式のAPI一覧表を**報告として**提示し、そのまま続行する。過不足の判断に迷った候補は仮定として注記する

   | API ID | メソッド | パス | 概要 | 対応要件 |
   |---|---|---|---|---|
   | API-001 | GET | /users | ユーザー一覧取得 | REQ-F-004 |

3. この時点で共通規約も決定する（既存コードベースがあれば `openapi-researcher` に現行の慣習を調査させ、それに合わせる。なければ業界標準の慣習を仮定として採用し注記する）
   - 認証方式 / 日時フォーマット / ページネーション方式 / エラーレスポンス形式と業務エラーコード体系 / 命名規則（ケバブ・スネーク・キャメル）
4. 決定した一覧を `tags`・`paths` の骨組みと `info.description` の共通規約としてファイルに反映する

### ステップ3: 不足情報の調査・質問

各APIの詳細を書くために足りない情報を3種類に分類して埋める。

1. **(a) コードベース調査で埋まる可能性がある項目**: 既存実装のルーティング・DTO・バリデーション・エラーハンドリング、現行APIとの互換性制約など → `openapi-researcher` エージェントに依頼する
2. **(b) Web調査で埋まる可能性がある項目**: 業界標準の慣習（ページネーション・エラー形式・認証フロー）、外部連携先APIの仕様など → `openapi-researcher` エージェントに依頼する
3. **(c) 本来ユーザーが決める項目**: ビジネスルール（重複時の挙動、権限の線引き、境界値の扱い）、公開範囲、互換性ポリシーなど → **ユーザーに質問しない**（質問は実行前確認で完了している）
   - 要件定義書・調査結果から最も合理的な仮定を置いて埋め、`description` に「（仮定）」と注記した上で `x-open-issues` に `OPEN-XXX` として記録する
   - 仮定を置く根拠すらない項目は、埋めずに `OPEN-XXX` のみ記録する

### ステップ4: 詳細記述

API一覧の各行について、記述レベルガイドの基準で詳細を書く。

1. 1つのAPIを書き終えるごとに、記述レベルガイドの「各操作ごと」チェックリストで自己監査する。未達の項目を残したまま次へ進まない（情報が得られない場合は `x-open-issues` に記録した上で進む）
2. スキーマはリクエスト用・レスポンス用を分離し、「スキーマごと」チェックリストを満たす
3. 共通化できるパラメータ・レスポンスは `components` に定義して `$ref` で参照する
4. example は現実的なダミー値にする（"string"・"test" を使わない）

### ステップ5: ID採番・整合性チェック

1. `x-api-id`（API-XXX）が全操作に一意に振られていることを確認する。**既存ファイルを更新する場合、既存の API-XXX・operationId は変更・再採番しない**（画面仕様書・実装・テストからの参照を壊さないため）。新規APIには欠番を避けて新しい連番を割り当てる
2. `x-requirements` の全IDが要件定義書に実在することを突き合わせて確認する。存在しないIDは書かない
3. 逆方向のカバレッジを確認する: 要件定義書のAPI関連の機能要件のうち、どのAPIからも参照されていないものを洗い出し、合理的に必要と判断できるAPIは追加（仮定として注記）し、判断が分かれるものは `x-open-issues` に記録して続行する
4. 対応する要件が存在しないAPIが必要になった場合は、`x-open-issues` に記録し、最終報告で `/requirements-specificater` での要件追加を案内する（要件定義書は変更しない）

### ステップ6: 検証・出力・サマリー

1. テンプレート由来の `# GUIDE:` コメントと、未使用のテンプレート雛形（`_(未記入)_` のプレースホルダー行）を成果物から削除する
2. YAMLとして構文が正しいこと・`$ref` の参照先がすべて存在することを確認する。リンター（`npx @redocly/cli lint` または `npx @stoplight/spectral-cli lint`）が利用可能なら実行し、エラーを解消する（利用できない場合はスキップし、その旨を最終報告に記載する。グローバルインストールなど環境を変更する対処はしない）
3. 出力先に保存する
4. ユーザーに以下をサマリーとして提示する
   - API一覧（API-XXX / メソッド / パス / 対応要件）
   - 要件カバレッジ（要件定義書のREQ-F-XXXのうち対応済み・未対応のもの）
   - **採用した仮定の一覧**（共通規約・ビジネスルール等で「（仮定）」と注記した項目。ユーザーの判断で差し替えるべきもの）
   - 未確定事項（`x-open-issues`）の一覧と、解決のために誰に何を確認すべきか
   - 記述レベルチェックリストの充足状況（未達があれば正直に列挙する）

## 中断・再開

- 作業は複数回のセッションに分けてよい。再実行時も質問は実行前確認の1回のみ
- 再実行時は出力先ファイルを読み込み、`x-open-issues` と `_(未記入)_` 箇所、チェックリスト未達のAPIから再開する
- 1回のセッションで全APIを書ききる必要はない。ユーザーが切り上げを指示した場合は、その時点の状態で保存し、残課題を `x-open-issues` にまとめて終了する

## 制約

- ユーザーへの質問は実行前確認（ステップ0）の1回のみ。以降は報告のみで完遂する（例外規定の破壊的操作を除く）
- 画面レイアウト・画面遷移・テーブル設計・サーバー内部実装を本書に書かない
- 要件定義書に存在しない要件ID（REQ-F-XXX）を発明しない。要件定義書自体の変更は行わず、`/requirements-specificater` に委ねる
- 既存の `x-api-id`・`operationId`・公開済みのパス/フィールド名を変更しない（破壊的変更。必要な場合は例外規定として実行前に必ず確認する）
- ビジネスルール・権限の線引きを、注記なしの確定事項として書かない（仮定で埋める場合は必ず「（仮定）」注記と `x-open-issues` 記録をセットにし、最終報告で開示する）
- 機密情報・認証情報をWeb調査に渡さない（`openapi-researcher` エージェントへの依頼内容にも含めない）
- 記述レベルガイドのチェックリストを満たさない状態を「完成」と報告しない（未達は必ずサマリーで開示する）
