# dependency-updater 成果物テンプレート

エージェント間で受け渡す成果物ファイルの必須構成を定義する。各エージェントはここに定義された見出し構成を**厳守**する（下流エージェントが見出しを頼りに読み取るため）。セクションを省略する場合は見出しを残して「該当なし」と書く。

配置先はすべて `<対象リポジトリ>/.claude/dependency-updater/<作業ID>/` 配下（[architecture.md](architecture.md) 参照）。`<作業ID>` は Dependabot PR なら `pr-<番号>`、手動指定なら `manual-<パッケージ名>`。

---

## 1. 影響調査レポート（impact-report.md）

出力者: `dep-researcher` / 読者: 全エージェント

```markdown
# 影響調査レポート: <パッケージ名> <更新前バージョン> → <更新後バージョン>

- 作成日: YYYY-MM-DD
- 入力: Dependabot PR #<番号> / 手動指定
- エコシステム: npm / maven / github-actions / その他
- 対象リポジトリ: <絶対パス>

## 1. 更新サマリー

| 項目 | 値 |
|---|---|
| パッケージ | |
| 更新前 → 更新後 | |
| 更新種別 | major / minor / patch |
| 判定 | 修正必要 / 修正不要 / 大規模移行（要ユーザー判断） |

（グループ化された PR で複数パッケージが更新される場合は、パッケージごとに行を分ける）

## 2. 変更点調査（リリースノート・移行ガイド）

（更新前→更新後のバージョン範囲に含まれる変更点のみ。1件1見出し。該当がなければ「該当なし」）

### BC-1: <変更点の概要>

- 種別: 破壊的変更 / 非推奨化 / 挙動変更 / セキュリティ修正
- 内容: （何がどう変わったか。旧API→新APIの対応）
- 出典: <URL>（確度: 公式ドキュメント・リリースノート / 一般記事 / 推測）

## 3. 影響箇所（コードベース突合）

| 影響ID | 対応変更点 | ファイル:行 | 使用内容 | 修正要否 |
|---|---|---|---|---|
| IMPACT-1 | BC-1 | | | 要修正 / 影響なし |

（「影響なし」の行も消し込みの根拠として残す。要修正の影響IDのみ以下に詳細を書く）

### IMPACT-1: <概要>

- 修正方針: （移行ガイドの推奨方法に基づく具体的な修正内容。dep-fixer が追加調査なしに着手できる具体度で）
- 根拠: （BC-N の出典・既存コードの箇所）

## 4. 検証コマンド

| 用途 | コマンド | 実行ディレクトリ | 根拠 |
|---|---|---|---|
| ビルド | | | （Makefile / package.json 等の定義箇所。実在確認できないものは「未確認」と明記） |
| lint | | | |
| テスト | | | |

## 5. 残課題

（調査で解決しなかった点。下流エージェントが仮定を置く必要がある箇所）
```

## 2. 修正レポート（fix-reports/fix-attempt<N>.md）

出力者: `dep-fixer` / 読者: dep-tester, dep-reviewer

`<N>` は修正作業の実行回数（1始まり。初回修正・テスト失敗対応・レビュー差し戻し対応で通し番号）。

```markdown
# 修正レポート (attempt <N>)

- 作成日時: YYYY-MM-DD HH:MM
- 入力: impact-report.md / test-reports/test-attempt<M>.md / review-reports/review-attempt<K>.md（該当するもののみ）
- 作業種別: 初回修正 / テスト失敗対応 / レビュー差し戻し対応

## 1. 対応サマリー

| 対象ID | 状態 |
|---|---|
| IMPACT-1 / FAIL-1 / REV-1 | 修正済み / 対象外（理由） |

## 2. 変更内容

（1対象1見出し）

### <対象ID>: <概要>

- 変更ファイル: <パス>
- 変更内容: （何をどう変えたか）
- 根拠: （impact-report の修正方針 / 移行ガイドの出典）

## 3. 失敗原因の分類（作業種別が「テスト失敗対応」のときのみ）

| 失敗ケース | 分類 | 対応 |
|---|---|---|
| FAIL-1 | 更新起因 / 既存不具合 / 環境起因 | 修正済み / 対象外（記録のみ） |

- 分類の根拠: （「更新起因」以外に分類した場合は、今回の更新と無関係と判断した根拠を必ず書く）

## 4. 検証

（ビルド確認など、自分で実行した検証コマンドと結果。テストスイート全体の実行は dep-tester の担当）

## 5. 仮定・申し送り

（置いた仮定、dep-tester / dep-reviewer に伝えるべきこと）
```

## 3. テスト結果レポート（test-reports/test-attempt<N>.md）

出力者: `dep-tester` / 読者: dep-fixer, dep-reviewer

`<N>` はテスト実行回数（1始まり）。

```markdown
# テスト結果 (attempt <N>)

- 実行日時: YYYY-MM-DD HH:MM
- 検証コマンドの出典: impact-report.md「4. 検証コマンド」

## 1. サマリー

| ステップ | コマンド | 結果 | 件数（成功/失敗/スキップ） |
|---|---|---|---|
| ビルド | | PASS / FAIL / ERROR / SKIP | |
| lint | | | |
| テスト | | | |

- 総合判定: PASS / FAIL / ERROR（実行自体が失敗） / SKIP（検証コマンドなし）

## 2. 失敗ケース

（失敗がなければ「該当なし」。1ケース1見出し）

### FAIL-1: <テスト名 / ビルドエラーの概要>

- ステップ: ビルド / lint / テスト
- ファイル: <パス>
- 期待値:
- 実際値:
- エラーメッセージ・スタックトレース抜粋:

  ```
  （先頭の原因箇所を中心に。全文は貼らない）
  ```

- 更新パッケージとの関連（推定）: （どの BC-N / IMPACT-N に関係しそうか。無関係に見える場合はその旨）

## 3. 特記事項

（環境の問題、更新前から失敗していた疑いのあるテスト、フレーキーな挙動など dep-fixer に伝えるべきこと）
```

## 4. レビューレポート（review-reports/review-attempt<N>.md）

出力者: `dep-reviewer` / 読者: dep-fixer（スキルが must-fix を振り分ける）

```markdown
# レビューレポート (attempt <N>)

- 作成日時: YYYY-MM-DD HH:MM
- レビュー対象: 変更差分 / impact-report.md / fix-reports/ / test-reports/test-attempt<M>.md
- 判定: 通過 / 要修正
- イテレーション: <N> / 上限

## 1. 指摘サマリー

| 指摘 | 分類 | 差し戻し先 |
|---|---|---|
| REV-1 | must-fix / suggestion / 更新起因外 | dep-fixer / （対象外） |

## 2. must-fix

（1指摘1見出し。dep-fixer が追加調査なしに着手できる具体度で書く）

### REV-1: <概要>

- 根拠: （impact-report の記述 / 移行ガイドの出典 / コード箇所）
- 修正対象ファイル: <パス>
- 修正内容: （何をどう変えるか）

## 3. suggestion（差し戻さない改善提案）

- REV-N: （内容と理由。最終報告に記録する）

## 4. 更新起因外の課題

- REV-N: （既存不具合・環境問題など、今回の更新と無関係と判断した課題。判断根拠を必ず書く。コード修正の対象外として最終報告に記録する）

## 5. 前回指摘の解消状況（2回目以降のみ）

| 前回指摘 | 状態 |
|---|---|
| REV-1 | 解消 / 未解消 / 部分的 |
```
