# dependency-updater アーキテクチャ

依存関係のバージョン更新（Dependabot PR / 手動指定）に伴う既存コードの追随修正を、「影響調査 → コード修正 → テスト → レビュー」のワークフローで自動化するプラグインの設計ドキュメント。

## 全体フロー

```
ステップ0: 実行前確認（メイン会話・唯一の質問タイミング）
    │
ステップ1: 準備（メイン会話）
    │  PRモード: gh pr view で更新内容取得 → gh pr checkout
    │  手動モード: パッケージ名 + 更新前→更新後バージョンを確定
    ▼
ステップ2: 影響調査 ─── dep-researcher ──▶ impact-report.md
    │  判定「修正不要」→ ステップ4へスキップ
    │  判定「大規模移行」→ 例外規定によりユーザー確認
    ▼
ステップ3: 修正 ─────── dep-fixer ──────▶ fix-reports/fix-attempt<N>.md
    ▼
ステップ4: テスト ───── dep-tester ─────▶ test-reports/test-attempt<N>.md
    │  FAIL/ERROR ──▶ ステップ5: 修正ループ（dep-fixer に差し戻し → 再テスト。
    │                 更新起因リトライ上限デフォルト3回）
    ▼ PASS
ステップ6: レビュー ─── dep-reviewer ───▶ review-reports/review-attempt<N>.md
    │  must-fix ──▶ dep-fixer → 再テスト → 再レビュー（往復上限2回固定）
    ▼ 通過
ステップ7: 完了報告（メイン会話。コミットは /git-commit-pusher へ案内）
```

## エージェント責務

| エージェント | 責務 | してはならないこと |
|---|---|---|
| メイン会話（スキル） | 実行前確認・PR 情報取得とチェックアウト・成果物の受け渡し・ループ判定・報告 | 対象リポジトリのソースコード編集、コミット・プッシュ |
| `dep-researcher` | リリースノート・移行ガイドの Web 調査、使用箇所のコードベース走査、影響箇所と修正方針の特定、検証コマンドの抽出 | コード編集、ビルド・テストの実行 |
| `dep-fixer` | impact-report / 差し戻し指摘に基づくコード修正（テストコードの追随修正を含む）、手動モード時の依存定義更新 | テストスイート全体の実行判定、テストの弱体化（期待値の緩和・skip・削除） |
| `dep-tester` | ビルド・lint・既存テストスイートの実行と結果レポート作成 | コードの修正、失敗原因の修正着手 |
| `dep-reviewer` | 変更差分・影響箇所の消し込み・テスト結果の3点レビュー、must-fix / suggestion の分類 | コード修正、テスト実行、「テストを緩めて通す」方向の指示 |

## 成果物ディレクトリ

```
<対象リポジトリ>/.claude/dependency-updater/<作業ID>/
├── impact-report.md                    # dep-researcher の出力
├── fix-reports/fix-attempt<N>.md       # dep-fixer の出力（通し番号）
├── test-reports/test-attempt<N>.md     # dep-tester の出力
└── review-reports/review-attempt<N>.md # dep-reviewer の出力
```

`<作業ID>` は `pr-<番号>`（Dependabot PR）または `manual-<パッケージ名>`（手動指定）。複数 PR を一括処理する場合も PR ごとにディレクトリを分け、1件ずつ順に処理する（ブランチを跨いで成果物が混ざらないようにするため）。

成果物の見出し構成は [artifact-templates.md](artifact-templates.md) に従う。

## ループと終了条件

### 修正ループ（ステップ5）

- テスト FAIL / ERROR → 失敗レポートを `dep-fixer` に渡して再修正 → 再テスト
- 失敗原因の分類は `dep-fixer` が修正レポートで行う:
  - **更新起因**（今回のバージョン更新に由来）→ 修正する（リトライカウント +1）
  - **既存不具合**（更新前から失敗していたと判断できる）→ 修正せず記録して判定から除外（カウント外）
  - **環境起因**（ツール未導入・環境設定の問題）→ 手順を記録し該当ステップを SKIP 扱い（カウント外）
- 打ち切り条件（いずれか）:
  - 更新起因リトライが上限（デフォルト3回）に到達 → 未解決として記録し、レビューをスキップしてステップ7へ
  - 同じ失敗が2回連続で修正されない → dep-fixer の申し送りに従い打ち切り

### レビュー往復（ステップ6）

- must-fix → `dep-fixer` に差し戻し → 再テスト → 再レビュー
- 往復上限は**2回固定**（実行前確認で変更しない。差し戻し再作業は修正ループのリトライカウント外）
- 上限到達時は残 must-fix を未解決の課題として記録しステップ7へ

## 設計判断

- **オーケストレーター＝メイン会話**: サブエージェントは AskUserQuestion を使えないため、実行前確認が成立するのはメイン会話のみ（code-implementer と同じ判断）
- **テストコードの修正も dep-fixer に集約**: バージョン更新では旧 API を使うテストコードの追随修正が本質的に発生するため、実装とテストを分担させない。テストの弱体化（期待値の緩和・skip・削除）は dep-reviewer が検査することで担保する
- **fix-planner 相当は置かない**: code-implementer と違い修正対象が「影響調査で特定済みの箇所 + その波及」に限定されるため、失敗分析は dep-fixer に内包して往復を減らす
- **「挙動を変えない最小修正」を原則とする**: バージョン更新の追随はリファクタリングの機会ではない。移行ガイドの推奨方法への置換に留め、計画外の変更混入は dep-reviewer が must-fix とする
- **既存不具合はスコープ外**: 更新前から失敗しているテスト・潜在バグは修正せず、判断根拠つきで最終報告に記録する（スコープ膨張の防止）
- **コミット・プッシュはしない**: `/git-commit-pusher` に委譲。Dependabot PR ブランチへ追いコミット＆プッシュすれば PR が更新される（人がコミットした PR を Dependabot は以後リベースしない仕様のため、その旨も報告に含める）
