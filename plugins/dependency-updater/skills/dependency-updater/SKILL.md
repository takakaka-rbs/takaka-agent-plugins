# /dependency-updater

依存パッケージのバージョン更新（Dependabot PR または手動指定）を入力に、更新によって既存コードで修正が必要になる箇所を調査し、修正・テスト・レビューまで進行管理するスキルです。

あなた（メイン会話）はこのワークフローのオーケストレーターです。調査・修正・テスト・レビューの実作業はすべてサブエージェントに委譲し、自分では PR 情報の取得とチェックアウト・成果物ファイルの受け渡し・順序制御・ループ判定・報告だけを行います。

## 実行モデル: 初回一括確認・以降は自動完遂

- ユーザーへの質問は**ステップ0「実行前確認」の1回だけ**にまとめる。作業開始後は承認・確認を求めず、調査結果・テスト結果は報告のみで最後まで実行する
- 作業中に判明した不明点は、調査で埋まらなければ合理的な仮定を置いて進める。採用した仮定は最終報告に明記する
- **例外 — 次の場合だけは実行前に必ず確認する**: 影響調査の判定が「大規模移行」（フレームワークのメジャー移行級の書き換えが必要）となった場合、未コミット変更がある状態での PR ブランチへのチェックアウト、既存ファイルの削除を伴う修正

## 前提と設計ドキュメント

- ワークフロー設計・エージェント責務・ループ終了条件: `${CLAUDE_PLUGIN_ROOT}/docs/architecture.md`
- 成果物（影響調査レポート・修正レポート・テスト結果・レビューレポート）の形式: `${CLAUDE_PLUGIN_ROOT}/docs/artifact-templates.md`

エージェントへの委譲時は、該当テンプレートの見出し構成に従うことを毎回指示してください。

## Usage

```
/dependency-updater [対象リポジトリのパス] [PR番号 | パッケージ名 更新前→更新後]
```

引数は省略可能です。省略した場合は実行前確認（ステップ0）で確認します。

例:

```
/dependency-updater 52                                # Dependabot PR #52 を処理
/dependency-updater 52 53 54                          # 複数PRを順に処理
/dependency-updater vue-router 4.3.0→4.4.0            # 手動指定（PRなし）
```

## 成果物の受け渡し

エージェント間の受け渡しは `<対象リポジトリ>/.claude/dependency-updater/<作業ID>/` 配下のファイルを正とします。`<作業ID>` は PR モードなら `pr-<番号>`、手動モードなら `manual-<パッケージ名>`。

```
.claude/dependency-updater/<作業ID>/
├── impact-report.md                    # dep-researcher の出力
├── fix-reports/fix-attempt<N>.md       # dep-fixer の出力（通し番号）
├── test-reports/test-attempt<N>.md     # dep-tester の出力
└── review-reports/review-attempt<N>.md # dep-reviewer の出力
```

各エージェントには常に「入力ファイルのパス」と「出力先パス」を明示して委譲します。作業ディレクトリが対象リポジトリの Git 管理に含まれる場合は、`.gitignore` への `.claude/dependency-updater/` 追記を初回に**報告**します（勝手にコミットしない）。

## 全体フロー

### ステップ0: 実行前確認（唯一の質問タイミング）

方針判断が必要な事項を、**最大4問までの一括質問として最初に1回だけ**確認する。引数・プロジェクトの文脈から自明な項目は聞かずに省略し、全項目が自明なら質問せずそのまま進む。

質問候補（該当する場合のみ）:

1. **対象リポジトリ**: カレントディレクトリが対象リポジトリと判別できない場合のみ
2. **更新対象**: 引数から自明でない場合。PR モードでは `gh pr list --label dependencies --state open` で Dependabot PR の候補一覧を提示して選ばせる（複数選択可）。PR がない・gh が使えない場合はパッケージ名と更新前→更新後バージョンの手動指定を受ける
3. **検証範囲**: 既存のビルド・lint・テストのうちどこまで実行するか（デフォルト提案: すべて。テストスイートの実行時間が長いことが既知の場合はその旨を添える）
4. **修正ループの最大リトライ回数**: デフォルト3回（更新起因の失敗に対する再修正の上限）

### ステップ1: 準備

1. **PR モード**: `gh --version` / `gh auth status` で gh CLI の利用可否を確認する（GitHub MCP が利用可能ならそちらを優先してよい）。`gh pr view <番号>` で PR のタイトル・本文（リリースノート抜粋）・変更ファイルから、更新パッケージ・更新前→更新後バージョン・エコシステムを読み取る
2. **PR モード**: 未コミット変更がないことを確認して `gh pr checkout <番号>` で PR ブランチに切り替える（未コミット変更がある場合は例外規定に従い確認する）
3. **手動モード**: 現在のブランチのまま作業する。更新後バージョンが未指定なら dep-researcher に最新安定版の特定を含めて委譲する
4. 成果物ディレクトリ `<作業ID>/` を作成する
5. **複数 PR の一括処理**: 1件ずつ順に処理する（PR ごとにステップ1〜7を完結させてから次へ。ブランチを跨ぐため並行処理しない）

### ステップ2: 影響調査（dep-researcher）

1. `dep-researcher` に対象リポジトリのパス・更新対象（パッケージ・バージョン・エコシステム・PR 本文）・出力先（`impact-report.md`）を渡して委譲する
2. 返ってきた要点（判定・要修正の影響箇所数・特記事項）を**報告として**提示する
3. 判定による分岐:
   - **修正必要** → ステップ3へ
   - **修正不要** → ステップ3をスキップしてステップ4（テスト）へ（更新自体の互換性をテストで確認する）
   - **大規模移行** → 例外規定に従い、調査結果の要約と移行規模を提示して続行可否をユーザーに確認する

### ステップ3: 修正（dep-fixer）

1. `dep-fixer` に影響調査レポートのパス・対応する IMPACT-N・出力先（`fix-reports/fix-attempt<N>.md`）を渡して委譲する。手動モードの場合は依存定義の更新も指示する
2. 完了報告（変更ファイル・検証結果・仮定・申し送り）を確認する。「調査漏れ」の申し送りがあった場合はレビュー時に dep-reviewer へ伝える

### ステップ4: テスト（dep-tester）

1. `dep-tester` に影響調査レポートのパス（検証コマンドの正）・attempt 番号・出力先（`test-reports/test-attempt<N>.md`）を渡して委譲する。2回目以降は前回レポートのパスも渡す
2. 総合判定による分岐:
   - **PASS** → ステップ6（レビュー）へ
   - **FAIL / ERROR** → ステップ5（修正ループ）へ
   - **SKIP**（検証コマンドなし）→ 検証できなかった旨を記録してステップ6へ（レビューで補完する）

### ステップ5: 修正ループ（dep-fixer → 再テスト）

1. `dep-fixer` に失敗レポート・影響調査レポートのパス・リトライ状況・出力先を渡して原因分類と再修正を委譲する
2. 修正レポートの分類に従って扱う:
   - **更新起因** → 修正済みの内容で再テストへ（リトライカウント +1）
   - **既存不具合** → 修正対象外として記録し、該当ケースを以後の判定から除外して続行（カウント外）
   - **環境起因** → ユーザーが行う手順を記録し、該当ステップを SKIP 扱いにして続行（カウント外）
3. 再テストは attempt を進めて実行する
4. **終了条件**(いずれかで修正ループを打ち切る):
   - 対象の失敗がすべて解消（PASS または除外）→ ステップ6へ
   - 更新起因リトライがステップ0の上限（デフォルト3回）に到達 → 未解決として記録し、ステップ6は実行せずステップ7へ
   - dep-fixer が打ち切りを推奨（修正の見込みなし・同じ失敗の繰り返し）→ 未解決として記録しステップ7へ

### ステップ6: レビュー（dep-reviewer によるレビューゲート）

1. `dep-reviewer` に対象リポジトリ・更新対象・影響調査レポート・修正レポート・最新テスト結果のパス・変更ファイル一覧・出力先（`review-reports/review-attempt<N>.md`）・イテレーション状況を渡して委譲する。レビュー対象は**変更差分・影響箇所の消し込み・テスト結果の3点**
2. 判定が**通過**（must-fix 0件）なら、suggestion・更新起因外の記録を持ってステップ7へ進む
3. must-fix がある場合は差し戻す:
   - 該当 REV-N を `dep-fixer` に委譲 → 再テスト（attempt を進める。FAIL したら通常どおりステップ5で収束させる）→ `dep-reviewer` に再レビューさせる（前回レポートのパスを渡す）
4. **レビュー往復の上限はデフォルト2回**（実行前確認で変更しない固定値。差し戻し再作業はステップ5のリトライカウントに数えない）。上限到達時点で残った must-fix は未解決の課題として記録し、ステップ7へ進む

### ステップ7: 完了報告

ユーザーに以下をサマリーとして提示する（複数 PR の場合は PR ごとに区切って報告する）。

- 更新対象（パッケージ・更新前→更新後・PR 番号）と判定
- 変更点（BC-N）と対応した影響箇所（IMPACT-N）の一覧・変更ファイル
- テスト結果（最終 attempt の判定と件数、レポートのパス）
- 修正ループの履歴（attempt 数・主な修正内容）
- レビュー結果（判定・must-fix の対応履歴・suggestion の一覧。レビュー未実施の場合はその理由）
- **採用した仮定の一覧**（調査・修正で置いたもの）
- 未解決の課題: 既存不具合（判断根拠つき）・環境起因（ユーザーが行う手順）・リトライ上限で残った失敗
- コミットは行っていないこと（コミット・プッシュは `/git-commit-pusher` に委ねる旨の案内。PR ブランチへ追いコミット＆プッシュすれば Dependabot PR が更新されること、人がコミットした PR を Dependabot は以後自動リベースしなくなることを添える）

## 中断・再開

- 作業は複数セッションに分けてよい。再実行時も質問は実行前確認の1回のみ
- 再実行時は `<作業ID>/` 配下の成果物（影響調査レポート・最後の attempt の各レポート）を読み込み、続きから再開する（影響調査が既にあれば作り直さない。対象ブランチの状態が大きく変わっている場合のみ dep-researcher に差分調査させる）

## 制約

- ユーザーへの質問は実行前確認（ステップ0）の1回のみ。以降は報告のみで完遂する（例外規定を除く）
- メイン会話（あなた）は対象リポジトリのソースコードを直接編集しない。調査・修正・テスト・レビューはすべて担当エージェントに委譲する（成果物ディレクトリの作成・`gh pr checkout` によるブランチ切替・軽微なファイル操作は可）
- git のコミット・プッシュをしない（`/git-commit-pusher` の担当。完了報告で案内する）。PR のマージ・クローズもしない
- 指定された更新以外の依存を追加・削除・更新しない
- 各エージェントの制約（researcher は編集しない、fixer はテストスイートを実行しない、tester は修正しない、reviewer は編集・実行しない）を跨ぐ指示を出さない
- テストが失敗したまま・must-fix が残ったまま「完了」と報告しない（未解決事項は必ずサマリーで開示する）
- 機密情報（認証情報・接続文字列）をエージェントへの指示・Web調査に含めない
