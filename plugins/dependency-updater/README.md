# dependency-updater

Dependabot が作成した依存関係更新 PR（または手動指定のパッケージ更新）を入力に、バージョン更新に伴う既存コードの追随修正を「影響調査 → コード修正 → テスト → レビュー」のワークフローで自動実行するプラグイン。

Dependabot は依存定義（package.json / pom.xml 等）の更新までしかやってくれない。破壊的変更・非推奨化・挙動変更に対する**既存コード側の追随修正**が本プラグインの担当範囲。

## 特徴

- **Dependabot PR 連携**: PR 番号を渡すだけで、PR 本文のリリースノート・変更ファイルから更新内容を読み取り、`gh pr checkout` で PR ブランチ上に追随修正を積む。複数 PR の一括処理（1件ずつ順次）にも対応
- **手動指定にも対応**: PR がなくても「vue-router 4.3.0→4.4.0」のようなパッケージ＋バージョン指定で動く（依存定義の更新も行う）
- **根拠ベースの影響調査**: リリースノート・移行ガイドの Web 調査（出典・確度つき）と、対象 API の使用箇所のコードベース走査を突き合わせ、影響箇所を消し込み表で管理する
- **挙動を変えない最小修正**: 移行ガイドの推奨方法への置換に徹し、リファクタリング等の計画外変更はレビューゲートで must-fix とする
- **テスト弱体化の検査**: 期待値の緩和・skip・削除で「通しただけ」の修正をレビューで検出する
- **質問は実行前確認の1回のみ**: 以降は修正ループ（上限3回）・レビュー往復（上限2回）まで自動で完遂する

## 前提

- 対象リポジトリが Git 管理されていること
- PR モードを使う場合: `gh` CLI が認証済みであること（または GitHub MCP が利用可能であること）
- 検証はリポジトリ**既存の**ビルド・lint・テストコマンド（Makefile / package.json scripts 等）を使う。新しいテストは書かない

## 使い方

```
/dependency-updater [対象リポジトリのパス] [PR番号 | パッケージ名 更新前→更新後]
```

例:

```
/dependency-updater 52                      # Dependabot PR #52 を処理
/dependency-updater 52 53 54                # 複数PRを順に処理
/dependency-updater vue-router 4.3.0→4.4.0  # 手動指定（PRなし）
/dependency-updater                         # 引数なし → 実行前確認で PR 候補一覧から選択
```

## ワークフロー

```
実行前確認（1回のみ） → 準備（PR取得・checkout） → 影響調査 → 修正 → テスト ⇄ 修正ループ → レビュー ⇄ 差し戻し → 完了報告
```

詳細は [docs/architecture.md](docs/architecture.md) を参照。

## 構成

| コンポーネント | 名前 | 役割 |
|---|---|---|
| スキル | `/dependency-updater` | ワークフロー全体のオーケストレーション |
| エージェント | `dep-researcher` | 変更点の Web 調査と影響箇所のコードベース突合（影響調査レポート） |
| エージェント | `dep-fixer` | 影響箇所の追随修正・失敗原因の分類・差し戻し対応 |
| エージェント | `dep-tester` | 既存ビルド・lint・テストスイートの実行と結果レポート |
| エージェント | `dep-reviewer` | テスト通過後のレビューゲート（差分・消し込み・テスト結果の3点レビュー） |

## 成果物

エージェント間の受け渡しは `<対象リポジトリ>/.claude/dependency-updater/<作業ID>/` 配下のマークダウンで行う（`pr-<番号>` / `manual-<パッケージ名>`）。形式は [docs/artifact-templates.md](docs/artifact-templates.md) に定義。

## やらないこと

- git のコミット・プッシュ（`/git-commit-pusher` に委譲。PR ブランチへ追いコミット＆プッシュすれば Dependabot PR が更新される）
- PR のマージ・クローズ
- 指定された更新以外の依存の追加・削除・更新
- 既存不具合（更新前から失敗しているテスト等）の修正 — 判断根拠つきで最終報告に記録するのみ
- 新しいテストの作成（既存テスト資産が通ることの確認まで）
